# 并行策略总览（DP/DDP/TP/PP/3D并行）

## 一句话结论
DP/DDP 通过 all-reduce 同步梯度适合小模型，TP 按张量切分适合大模型单层，PP 按层切分适合超深模型；3D 并行（DP×TP×PP）通过组合策略突破单卡显存瓶颈，但通信开销随规模增长，选择策略需权衡显存、通信、计算效率。

## 核心定义/公式

### 1. Data Parallelism (DP/DDP)

**数据并行核心公式**：
```
# 梯度同步（All-Reduce）
grad_global = all_reduce(grad_local) / world_size

# 参数更新
θ = θ - lr * grad_global
```

**关键配置**：
- `world_size`: GPU 数量（如 8 卡 = 8）
- `batch_size_per_gpu`: 单卡 batch size
- `global_batch_size = batch_size_per_gpu × world_size`
- 通信量：`2 × model_size`（梯度 all-reduce，发送+接收）

**DDP vs DP（PyTorch）**：
```python
# DP（单进程多线程，已被废弃）
model = nn.DataParallel(model, device_ids=[0,1,2,3])

# DDP（多进程，推荐）
model = DDP(model, device_ids=[local_rank])
# 优势：1）避免 GIL 锁 2）进程级隔离 3）通信与计算重叠
```

### 2. Tensor Parallelism (TP)

**张量切分原理**（以线性层为例）：
```python
# 原始：Y = XW, W∈[d_in, d_out]
# TP 切分：W = [W_1, W_2, ..., W_n]，每个 GPU 持有 W_i∈[d_in, d_out/n]

# Forward
# Step 1: All-Gather X（每个 GPU 获得完整输入）
X_full = all_gather(X_local)  # [batch, seq, d_in]

# Step 2: 本地计算
Y_local = X_full @ W_local  # [batch, seq, d_out/n]

# Step 3: All-Reduce 汇聚（每个 GPU 获得完整输出）
Y_full = all_reduce(Y_local)  # [batch, seq, d_out]

# 或使用 Reduce-Scatter（节省显存）
Y_local = reduce_scatter(Y_local)  # [batch, seq, d_out/n]
```

**TP 在 Transformer 中的应用**：
```python
# Attention 层
Q = X @ W_q  # TP 切分 W_q，按 head 切分
K = X @ W_k
V = X @ W_v
# Q/K/V 每个 GPU 持有部分 head，计算后 all-reduce

# FFN 层
hidden = gelu(X @ W_1)  # W_1 按列切分
output = hidden @ W_2   # W_2 按行切分
# 中间 all-reduce 一次
```

**关键参数**：
- `tp_degree`: TP 并行度（如 2/4/8）
- 通信量：`2 × batch × seq_len × d_model × tp_degree`（每个 TP 层）

### 3. Pipeline Parallelism (PP)

**流水并行原理**：
```python
# 模型按层切分
# GPU 0: Layer 0-5
# GPU 1: Layer 6-11
# GPU 2: Layer 12-17
# GPU 3: Layer 18-23

# Micro-batch 流水执行
# 时间步 | GPU0 | GPU1 | GPU2 | GPU3
#   t0  | mb0  |      |      |      
#   t1  | mb1  | mb0  |      |      
#   t2  | mb2  | mb1  | mb0  |      
#   t3  | mb3  | mb2  | mb1  | mb0  
#   t4  |      | mb3  | mb2  | mb1  
# ...
```

**Bubble 计算公式**：
```
Bubble 比例 = (pp_degree - 1) / (pp_degree + num_micro_batches - 1)

# 示例：pp_degree=4, num_micro_batches=8
# Bubble = (4-1) / (4+8-1) = 3/11 ≈ 27%
```

**关键配置**：
```python
# Megatron-LM 配置示例
num_micro_batches = global_batch_size // (batch_size_per_gpu * dp_degree)
# 微批次数 = 全局批次 / (单卡批次 × DP 度)
# 微批次数越多，bubble 越小，但延迟越高
```

### 4. 3D 并行组合

**组合策略**：
```
3D 并行 = DP × TP × PP

# 示例：70B 模型，8 卡 A100（80GB）
# 配置：DP=2, TP=2, PP=2
# - DP=2: 2 组数据并行
# - TP=2: 每组内 2 卡做张量并行
# - PP=2: 每组内 2 卡做流水并行

# 总 GPU = DP × TP × PP = 2 × 2 × 2 = 8 ✓
```

**显存分解**：
```
单卡显存占用 = 模型参数 + 梯度 + 优化器状态 + 激活值

# TP 切分：模型参数、梯度、优化器状态都按 TP 度切分
# PP 切分：每卡只存储部分层的参数
# DP：不节省显存，每卡存储完整模型（但可配合 ZeRO）

# 示例：70B 模型
# - FP16 参数：70B × 2B = 140GB
# - TP=8: 140GB / 8 = 17.5GB/卡
# - 再加激活值、梯度等，单卡约 30-40GB
```

## 为什么（2-3 个因果链）

### 1. 为什么需要不同并行策略？

**因果链：模型规模 → 显存瓶颈 → 需要切分策略**

- **现象**：大模型训练超出单卡显存（如 GPT-4 约万亿参数，A100 80GB 显存远不够）
- **根因**：显存限制 ← 参数 + 梯度 + 优化器状态 + 激活值共同占用
- **结果**：必须切分模型或优化显存使用

**DP 的局限**：
- DP 只复制模型，每卡存储完整模型
- 无法突破单卡显存上限
- 通信开销随 GPU 数增长（all-reduce 梯度）

**TP/PP 的优势**：
- TP 切分单层参数，突破单层显存瓶颈
- PP 切分层堆叠，突破深度显存瓶颈
- 两者配合可训练任意规模模型

### 2. 为什么 TP 通信频繁但仍被使用？

**因果链：切分粒度 → 通信开销 → 计算重叠**

- **现象**：TP 每个 Transformer 层都需要 all-reduce/all-gather，看似通信开销大
- **根因**：
  - 通信发生在层间，计算发生在层内
  - 现代网络带宽高（InfiniBand 200Gbps+）
  - 通信与计算可部分重叠（发送上一批结果时计算下一批）
- **结果**：TP 在大模型（单层参数大）时收益 > 代价

**关键洞察**：
```python
# TP 通信开销 vs 计算开销
# 计算量：2 × d_model² × batch × seq_len（矩阵乘法）
# 通信量：2 × d_model × batch × seq_len × tp_degree（all-reduce）

# 计算通信比 = d_model / tp_degree
# d_model 越大，计算占比越高，通信相对可接受
# 如 GPT-3: d_model=12288, tp_degree=8 → 计算通信比=1536
```

### 3. 为什么 PP 存在 Bubble 但仍被使用？

**因果链：流水执行 → Bubble 开销 → 微批次优化**

- **现象**：PP 开始和结束阶段存在 Bubble（部分 GPU 空闲）
- **根因**：
  - 流水启动：前几个微批次只能填满部分 GPU
  - 流水结束：后几个微批次只能排空部分 GPU
  - Bubble 比例 = (PP 度 - 1) / (微批次数 + PP 度 - 1)
- **结果**：
  - 增加微批次数可降低 Bubble 比例
  - 但增加延迟（需等所有微批次完成）
  - 适用场景：训练大 batch（微批次数多）时 Bubble 影响小

**Bubble 优化**：
```
# 1F1B 策略（One Forward One Backward）
# 交错执行 forward 和 backward，减少 Bubble
# 时间步 | GPU0 | GPU1 | GPU2 | GPU3
#   t0  | F0   |      |      |      
#   t1  | F1   | F0   |      |      
#   t2  | F2   | F1   | F0   |      
#   t3  | B0   | F2   | F1   | F0   # 开始 backward
#   t4  | F3   | B0   | F2   | F1   
#   t5  | B1   | F3   | B0   | F2   
# ...
```

## 怎么做（可落地步骤）

### 标准做法

#### 1. 选择并行策略的决策树

```
Step 1: 估算显存需求
  单卡显存需求 = 模型参数 + 梯度 + 优化器状态 + 激活值
  
  # FP16 混合精度训练
  参数：model_size × 2B
  梯度：model_size × 2B
  优化器状态（AdamW）：model_size × 12B（FP32 参数 + momentum + variance）
  激活值：≈ batch × seq_len × d_model × num_layers × 2B

Step 2: 判断是否需要模型并行
  if 单卡显存需求 > GPU 显存上限:
      必须使用模型并行（TP/PP）
  else:
      可用 DP/DDP（简单高效）

Step 3: 选择 TP vs PP
  if 模型宽度大（d_model 大，如 > 10k）:
      优先 TP（切分单层参数，通信开销可接受）
  elif 模型深度大（层数多）:
      优先 PP（切分层堆叠，通信少）
  else:
      TP+PP 组合（平衡通信与计算）

Step 4: 调整 DP 度
  DP 度 = 总 GPU 数 / (TP 度 × PP 度)
  
  if DP 度 >= 1:
      全局批次 = 单卡批次 × DP 度
      可用 DDP 进一步提升吞吐
  else:
      无数据并行，纯模型并行
```

#### PyTorch DDP 基础实现

```python
import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data import DataLoader, DistributedSampler

def setup_ddp(rank, world_size):
    """初始化 DDP 环境"""
    dist.init_process_group(
        backend='nccl',  # 使用 NCCL 后端（GPU）
        init_method='tcp://localhost:12355',
        rank=rank,
        world_size=world_size
    )
    torch.cuda.set_device(rank)

def cleanup_ddp():
    """清理 DDP 环境"""
    dist.destroy_process_group()

def train_ddp(rank, world_size):
    setup_ddp(rank, world_size)
    
    # 1. 创建模型并移至 GPU
    model = MyModel().cuda(rank)
    
    # 2. 用 DDP 包装模型
    ddp_model = DDP(
        model, 
        device_ids=[rank],
        output_device=rank,
        find_unused_parameters=False  # 加速：确认无未使用参数
    )
    
    # 3. 数据加载器（分布式采样）
    dataset = MyDataset()
    sampler = DistributedSampler(
        dataset,
        num_replicas=world_size,
        rank=rank,
        shuffle=True
    )
    dataloader = DataLoader(
        dataset,
        batch_size=32,
        sampler=sampler,
        num_workers=4,
        pin_memory=True  # 加速数据传输
    )
    
    # 4. 优化器（DDP 会自动同步梯度）
    optimizer = torch.optim.AdamW(ddp_model.parameters(), lr=1e-4)
    
    # 5. 训练循环
    for epoch in range(num_epochs):
        sampler.set_epoch(epoch)  # 确保 epoch 间 shuffle
        for batch_idx, (data, target) in enumerate(dataloader):
            data, target = data.cuda(rank), target.cuda(rank)
            
            optimizer.zero_grad()
            output = ddp_model(data)
            loss = criterion(output, target)
            loss.backward()  # DDP 自动同步梯度
            
            # 梯度裁剪（可选）
            torch.nn.utils.clip_grad_norm_(ddp_model.parameters(), max_norm=1.0)
            
            optimizer.step()
            
            if rank == 0 and batch_idx % 100 == 0:
                print(f'Epoch {epoch}, Batch {batch_idx}, Loss: {loss.item():.4f}')
    
    cleanup_ddp()

# 启动多进程
if __name__ == "__main__":
    world_size = torch.cuda.device_count()
    torch.multiprocessing.spawn(train_ddp, args=(world_size,), nprocs=world_size)
```

#### Megatron-LM TP 实现核心

```python
import torch
import torch.distributed as dist

class ColumnParallelLinear(torch.nn.Module):
    """列并行线性层（TP 切分输出维度）"""
    
    def __init__(self, in_features, out_features, world_size, rank):
        super().__init__()
        self.in_features = in_features
        self.out_features = out_features
        self.world_size = world_size
        self.rank = rank
        
        # 切分输出维度
        assert out_features % world_size == 0
        self.out_features_per_partition = out_features // world_size
        
        # 每个 GPU 只持有部分权重
        self.weight = torch.nn.Parameter(
            torch.empty(self.out_features_per_partition, in_features)
        )
        torch.nn.init.kaiming_uniform_(self.weight)
    
    def forward(self, x):
        # x: [batch, seq_len, in_features]
        # 本地计算：[batch, seq_len, out_features_per_partition]
        output = torch.nn.functional.linear(x, self.weight)
        
        # All-Reduce 汇聚完整输出
        dist.all_reduce(output, op=dist.ReduceOp.SUM)
        
        return output  # [batch, seq_len, out_features]

class RowParallelLinear(torch.nn.Module):
    """行并行线性层（TP 切分输入维度）"""
    
    def __init__(self, in_features, out_features, world_size, rank):
        super().__init__()
        self.in_features = in_features
        self.out_features = out_features
        self.world_size = world_size
        self.rank = rank
        
        # 切分输入维度
        assert in_features % world_size == 0
        self.in_features_per_partition = in_features // world_size
        
        # 每个 GPU 只持有部分权重
        self.weight = torch.nn.Parameter(
            torch.empty(out_features, self.in_features_per_partition)
        )
        torch.nn.init.kaiming_uniform_(self.weight)
    
    def forward(self, x):
        # x: [batch, seq_len, in_features_per_partition]
        # 本地计算：[batch, seq_len, out_features]
        output = torch.nn.functional.linear(x, self.weight)
        
        # All-Reduce 汇聚完整输出
        dist.all_reduce(output, op=dist.ReduceOp.SUM)
        
        return output  # [batch, seq_len, out_features]
```

## 权衡分析

| 并行策略 | 收益 | 代价 | 适用边界 |
|---------|------|------|----------|
| **DP/DDP** | 1. 实现简单<br>2. 通信开销可控（仅梯度同步）<br>3. 线性加速比（理想情况） | 1. 无法突破单卡显存瓶颈<br>2. 每卡存储完整模型副本<br>3. 全局批次受限（影响收敛） | 模型能装入单卡显存（< 10B 参数 @ 80GB） |
| **TP** | 1. 突破单层显存瓶颈<br>2. 适合大宽度模型<br>3. 通信可被计算重叠 | 1. 每个 Transformer 层都需通信<br>2. 通信量随 TP 度增长<br>3. 要求高带宽网络 | 单层参数大（d_model > 10k），GPU 间带宽高（InfiniBand） |
| **PP** | 1. 突破深度显存瓶颈<br>2. 通信量小（仅层间传递激活值）<br>3. 适合深层模型 | 1. Bubble 开销（流水启动/结束）<br>2. 层间负载不均（需精细划分）<br>3. 增加延迟（微批次串行） | 模型深度大（层数 > 48），批次大（微批次数多） |
| **3D 并行** | 1. 训练任意规模模型<br>2. 平衡显存与通信<br>3. 灵活配置（适应不同集群） | 1. 配置复杂（需调优 DP/TP/PP 度）<br>2. 通信模式复杂（多种原语组合）<br>3. 调试困难（故障定位难） | 超大模型（> 100B 参数），大规模集群（> 64 GPU） |
| **ZeRO + DP** | 1. 节省显存（切分优化器状态/梯度/参数）<br>2. 保持 DP 简单性<br>3. 可与 TP/PP 组合 | 1. 增加通信量（参数 all-gather）<br>2. ZeRO-3 通信开销显著<br>3. 对网络带宽敏感 | 中等模型（10-70B），网络带宽充足 |

## 高频追问（至少 5 个）

### 1. Q: DP 和 DDP 有什么区别？为什么推荐用 DDP？

**A:**
- **DP（DataParallel）**：单进程多线程，受 GIL 锁限制，性能差
- **DDP（DistributedDataParallel）**：多进程，进程级隔离，无 GIL 锁
- **DDP 优势**：
  1. 通信与计算重叠（梯度计算时同步上一批梯度）
  2. 进程隔离，稳定性高
  3. 支持模型并行等高级特性
- PyTorch 官方已废弃 DP，推荐 DDP

### 2. Q: TP 的 all-reduce 和 all-gather 有什么区别？分别用在哪些场景？

**A:**
- **All-Reduce**：所有 GPU 获得聚合后的完整结果（如求和后的张量）
  - 用途：TP 中汇聚切分后的输出（如 Attention 输出、FFN 输出）
  - 通信量：`N × 数据量`（N 为 GPU 数）
  
- **All-Gather**：所有 GPU 获得**拼接**后的完整张量
  - 用途：TP 中收集完整输入（如 Transformer 输入）
  - 示例：`X_local=[x1,x2], X_local=[x3,x4] → X_full=[x1,x2,x3,x4]`（所有 GPU 都有）
  - 通信量：`N × 数据量`

- **Reduce-Scatter**：All-Reduce 的逆操作，每个 GPU 获得聚合后的一部分
  - 用途：节省显存，每个 GPU 只保留部分输出
  - 示例：`Y_local=[y1,y2], Y_local=[y3,y4] → all-reduce → Y_full → reduce-scatter → Y_local=[y1', y2']`（每个 GPU 只有一部分）

### 3. Q: PP 的 Bubble 如何优化？1F1B 是什么？

**A:**
- **Bubble 成因**：
  - 流水启动：前几个微批次只能填满部分 GPU
  - 流水结束：后几个微批次只能排空部分 GPU
  - Bubble 比例：`(PP 度 - 1) / (微批次数 + PP 度 - 1)`

- **优化策略**：
  1. **增加微批次数**：降低 Bubble 比例
     - 示例：PP=4, 微批次数=4 → Bubble=3/6=50%
     - 示例：PP=4, 微批次数=16 → Bubble=3/18=17%
  
  2. **1F1B 策略**：交错执行 forward 和 backward
     - 原始：FFF...F → BBB...B（先全做 forward，再全做 backward）
     - 1F1B：F → F → F → B → F → B → F → B...（交错执行）
     - 优势：减少峰值显存（不需要存储所有前向激活值）
  
  3. **Interleaved Pipeline**：多个流水段循环执行
     - 每个 GPU 持有多段层（如 GPU0 持有 Layer0-5 和 Layer12-17）
     - 减少流水空隙

### 4. Q: 3D 并行中，DP/TP/PP 的配置顺序有什么讲究？

**A:**
- **推荐配置顺序**：先确定 TP/PP（模型并行），再确定 DP（数据并行）
  
- **TP 优先原则**：
  - TP 度首先由单层显存需求决定
  - 单卡显存 < 单层参数 → 必须用 TP
  - 计算公式：`TP 度 ≥ 单层参数 / 单卡显存`
  
- **PP 次优先**：
  - PP 度由总层数和显存分布决定
  - 尽量均匀分布层数，避免负载不均
  - 示例：96 层模型，PP=4 → 每个 GPU 24 层
  
- **DP 最后确定**：
  - DP 度 = 总 GPU 数 / (TP 度 × PP 度)
  - DP 度决定全局批次大小
  - DP 度太小 → 全局批次小 → 影响收敛

- **实际配置示例**：
  ```
  GPT-3 175B，512 张 A100
  
  Step 1: 估算 TP 度
  单层参数 ≈ 12288² × 4 × 2B = 1.2GB（FP16）
  单卡显存上限 = 80GB
  留给激活值、梯度等 ≈ 40GB
  TP 度 ≈ 175B × 2B / 40GB = 8.75 → TP=8
  
  Step 2: 确定 PP 度
  总 GPU = 512，TP=8
  剩余 GPU = 512 / 8 = 64
  PP 度 = 2（经验值，平衡通信与计算）
  
  Step 3: 确定 DP 度
  DP 度 = 512 / (8 × 2) = 32
  
  最终配置：DP=32, TP=8, PP=2
  ```

### 5. Q: 实际落地时，通信瓶颈一般卡在哪些环节？如何定位？

**A:**
- **主要瓶颈环节**：
  
  1. **梯度 All-Reduce（DP/DDP）**
     - 每个迭代结束同步梯度
     - 通信量：`2 × 模型参数量`
     - 瓶颈：模型越大，通信越慢
  
  2. **TP 层间 All-Reduce**
     - 每个 Transformer 层都需同步
     - 通信量：`2 × batch × seq_len × d_model × TP 度`
     - 瓶颈：序列长、TP 度大时通信开销大
  
  3. **PP 层间激活值传递**
     - 微批次间的点对点通信
     - 通信量：`batch × seq_len × d_model`
     - 瓶颈：批次大、序列长时延迟高
  
  4. **ZeRO 参数 All-Gather**
     - ZeRO-3 每层都需收集完整参数
     - 通信量：`模型参数量 × 通信次数`
     - 瓶颈：层数多、模型大时通信频繁

- **定位方法**：

- **优化策略**：
  - 使用高带宽网络（InfiniBand 200Gbps+）
  - 启用通信与计算重叠（`overlap_comm=True`）

  - 使用梯度压缩（如 FP16 → BF16 → INT8）

### 6. Q: 如何选择 ZeRO 的三个 stage？各有什么适用场景？

**A:**
- **ZeRO-1**：仅切分优化器状态
  - 节省显存：`优化器状态 / DP 度`（约 75% 节省）
  - 通信开销：小（仅优化器更新时通信）
  - 适用：显存稍微不够，但通信带宽有限

- **ZeRO-2**：切分优化器状态 + 梯度
  - 节省显存：`(优化器状态 + 梯度) / DP 度`（约 87.5% 节省）
  - 通信开销：中等（梯度 reduce-scatter）
  - 适用：中等模型（10-30B），需平衡显存与通信

- **ZeRO-3**：切分优化器状态 + 梯度 + 参数
  - 节省显存：`(优化器状态 + 梯度 + 参数) / DP 度`（约 94% 节省）
  - 通信开销：大（每个层都需 all-gather 参数）
  - 适用：超大模型（> 70B），显存极度紧张

- **选择建议**：
  ```
  if 显存充足:
      不用 ZeRO 或 ZeRO-1（通信开销最小）
  elif 显存轻微不足:
      ZeRO-1
  elif 显存中度不足:
      ZeRO-2
  else:  # 显存极度不足
      ZeRO-3 + CPU Offload（最后手段）
  ```

### 7. Q: 多机训练时，为什么容易抖动？如何优化？

**A:**
- **抖动原因**：
  
  1. **网络带宽不均**
     - 跨机通信带宽 < 机内通信（如以太网 < NVLink）
     - 多机同步时，最慢的节点拖慢整体
  
  2. **负载不均**
     - 不同节点的 GPU 性能差异（温度、频率）
     - 数据加载速度不同
  
  3. **网络抖动**
     - 丢包、延迟波动
     - InfiniBand 链路抖动

- **优化策略**：
  
  1. **网络优化**
     ```bash
     # 启用 NCCL 优化
     export NCCL_IB_DISABLE=0          # 启用 InfiniBand
     export NCCL_IB_HCA=mlx5           # 指定网卡
     export NCCL_SOCKET_IFNAME=eth0    # 指定以太网接口
     export NCCL_DEBUG=INFO            # 调试信息
     ```
  
  2. **负载均衡**
     - 确保每个节点的 GPU 数量一致
     - 使用 `DistributedSampler` 均匀分配数据
     - 监控各节点性能，调整批次大小
  
  3. **容错机制**
     ```python
     # 设置超时时间
     dist.init_process_group(timeout=datetime.timedelta(seconds=3600))
     
     # 心跳检测
     def heartbeat_check():
         if dist.is_initialized():
             # 定期广播心跳信号
             torch.distributed.broadcast(torch.tensor([1]), src=0)
     ```

### 8. Q: DP 和 DDP 在实现细节上有什么差异？梯度同步时机有何不同？

**A:**
- **DP（DataParallel）梯度同步**：
  - **时机**：`backward()` 结束后，立即 all-reduce
  - **方式**：主线程协调，多线程同步
  - **瓶颈**：GIL 锁限制，无法与计算重叠
  - **流程**：
    ```
    Forward → Backward → All-Reduce → Optimizer Step
    （串行，无法重叠）
    ```

- **DDP（DistributedDataParallel）梯度同步**：
  - **时机**：`backward()` 过程中，逐层异步同步
  - **方式**：每个进程独立，NCCL 异步通信
  - **优势**：通信与计算重叠
  - **流程**：
    ```
    Layer 1 Forward → Layer 1 Backward → 启动梯度同步（异步）
                                              ↓
    Layer 2 Forward → Layer 2 Backward ← 等待同步完成
    （并行，通信与计算重叠）
    ```

- **DDP 实现细节**：
  ```python
  # DDP 内部机制（简化版）
  class DDP:
      def __init__(self, module):
          self.module = module
          # 注册 backward hook
          for param in module.parameters():
              param.register_backward_hook(self._sync_gradient)
      
      def _sync_gradient(self, param, grad_input, grad_output):
          # 每层 backward 时异步启动 all-reduce
          if not self._sync_launched:
              self._sync_handle = dist.all_reduce(
                  grad_output, 
                  async_op=True
              )
              self._sync_launched = True
      
      def forward(self, x):
          output = self.module(x)
          self._sync_launched = False
          return output
  ```

### 9. Q: 如果模型单层能装入显存，但整体太大，应该优先用 TP 还是 PP？

**A:**
- **优先 PP**，原因：
  
  1. **通信开销更低**
     - TP：每个 Transformer 层都需 all-reduce（通信频繁）
     - PP：仅在层间传递激活值（通信少）
  
  2. **显存节省更明显**
     - TP：仅切分单层参数
     - PP：切分所有层，每个 GPU 只存储部分层
  
  3. **网络要求更低**
     - TP：要求高带宽低延迟（如 NVLink）
     - PP：可用普通以太网（如 100Gbps 以太网）

- **示例**：
  ```
  模型：100 层，每层 5GB 显存（单层能装入 80GB A100）
  总显存需求：100 × 5GB = 500GB
  
  方案 1：PP=8
  - 每个 GPU 存储 12-13 层
  - 显存：65GB（有余量）
  - 通信：层间传递激活值，每层约 10MB
  - 总通信：100 × 10MB = 1GB（可接受）
  
  方案 2：TP=8
  - 每个 GPU 存储完整模型的 1/8
  - 显存：500GB / 8 = 62.5GB
  - 通信：每个 Transformer 层 all-reduce
  - 总通信：100 层 × 通信量（较大）
  - 但单层 5GB < 80GB，TP 无必要
  ```

- **例外情况**：
  - 如果模型宽度极大（如 d_model > 20k），单层参数 > 单卡显存 → 必须用 TP
  - 如果网络带宽极高（如 NVLink 600GB/s），TP 开销可忽略 → 可用 TP

### 10. Q: 3D 并行中，如何避免不同 GPU 间的负载不均？

**A:**
- **负载不均原因**：
  
  1. **PP 层分布不均**
     - 不同层计算量不同（如 Attention vs FFN）
     - 前几层计算快，后几层计算慢
  
  2. **TP 切分不均**
     - head 数不能被 TP 度整除
     - 矩阵维度不能被切分
  
  3. **数据分布不均**
     - 不同样本长度不同（packing 后仍不均）
     - 不同 GPU 的批次大小不同

- **优化策略**：
  
  1. **PP 负载均衡**
     ```python
     # 方法 1：按计算量分配层
     # 不均匀分配：GPU 0 分配 20 层，GPU 1 分配 28 层
     # 根据每层计算量（FLOPs）动态调整
     
     # 方法 2：Interleaved Pipeline
     # 每个 GPU 持有多段层，循环执行
     # GPU 0: Layer 0-11, 48-59
     # GPU 1: Layer 12-23, 60-71
     # GPU 2: Layer 24-35, 72-83
     # GPU 3: Layer 36-47, 84-95
     ```
  
  2. **TP 切分对齐**
     ```python
     # 确保 head 数能被 TP 度整除
     num_heads = 96
     tp_degree = 8
     assert num_heads % tp_degree == 0  # 96 % 8 = 0 ✓
     
     # 确保隐藏维度能被切分
     hidden_size = 12288
     assert hidden_size % tp_degree == 0  # 12288 % 8 = 0 ✓
     ```
  
  3. **动态批次调整**
     ```python
     # 监控每个 GPU 的计算时间
     gpu_compute_times = [get_compute_time(rank) for rank in range(world_size)]
     
     # 调整批次大小
     if max(gpu_compute_times) / min(gpu_compute_times) > 1.2:
         # 负载不均 > 20%，调整批次
         adjust_batch_size()
     ```
  
  4. **使用 Profiler 定位瓶颈**
     ```python
     # 找出最慢的 GPU
     with torch.profiler.profile() as prof:
         train_step()
     
     # 分析每个 GPU 的计算时间
     for rank in range(world_size):
         print(f"GPU {rank}: {prof.get_compute_time(rank)}ms")
     ```

## 常见错误（至少 3 个）

### 1. 错误：TP 度设置过大，通信开销激增

**错误做法**：
```python
# 错误：TP=32，通信开销过大
# 假设模型 70B，d_model=12288
tp_degree = 32  # ❌ 通信量 = 2 × batch × seq_len × d_model × 32
```

**正确做法**：
```python
# 正确：根据单层显存需求设置 TP 度
# 单层参数 = d_model² × 4 × 2B（FP16 Attention + FFN）
# 单层参数 ≈ 12288² × 4 × 2B = 1.2GB

# 如果单卡显存 80GB，单层能装入
# TP 度 = 1（无需 TP）

# 如果单层参数 > 单卡显存
# TP 度 = ceil(单层参数 / 单卡显存)
tp_degree = max(1, ceil(layer_params / gpu_memory))
```

**原因**：
- TP 通信量与 TP 度成正比
- TP 度过大 → 通信时间 > 计算时间 → 性能下降
- 一般 TP 度不超过 8（经验值）

### 2. 错误：PP 微批次数过少，Bubble 比例过高

**错误做法**：
```python
# 错误：PP=4, 微批次数=4
# Bubble 比例 = (4-1) / (4+4-1) = 3/7 ≈ 43%
pipeline_parallel_size = 4
num_micro_batches = 4  # ❌ Bubble 太大
```

**正确做法**：
```python
# 正确：增加微批次数
pipeline_parallel_size = 4
num_micro_batches = 16  # ✓
# Bubble 比例 = (4-1) / (16+4-1) = 3/19 ≈ 16%

# 经验公式：微批次数 ≥ PP 度 × 4
num_micro_batches = max(PP_degree * 4, global_batch_size // (micro_batch_size * DP_degree))
```

**原因**：
- Bubble 比例 = (PP 度 - 1) / (微批次数 + PP 度 - 1)
- 微批次数越少，Bubble 越大
- Bubble 过大 → GPU 空闲率高 → 训练效率下降

### 3. 错误：DP/TP/PP 配置导致 DP 度为 0

**错误做法**：
```python
# 错误：总 GPU 数不足，DP 度计算错误
total_gpus = 8
tp_degree = 4
pp_degree = 2
dp_degree = total_gpus / (tp_degree * pp_degree)  # 8 / (4*2) = 1 ✓

# 但如果配置错误
total_gpus = 8
tp_degree = 4
pp_degree = 4
dp_degree = 8 / (4 * 4) = 0.5  # ❌ 不是整数，无法配置
```

**正确做法**：
```python
# 正确：确保 DP 度 ≥ 1 且为整数
def validate_3d_config(total_gpus, tp_degree, pp_degree):
    if total_gpus % (tp_degree * pp_degree) != 0:
        raise ValueError(f"总 GPU 数 {total_gpus} 必须能被 TP×PP={tp_degree*pp_degree} 整除")
    
    dp_degree = total_gpus // (tp_degree * pp_degree)
    
    if dp_degree < 1:
        raise ValueError(f"DP 度 {dp_degree} < 1，请增加 GPU 数或减少 TP/PP 度")
    
    return dp_degree

# 示例
total_gpus = 128
tp_degree = 8
pp_degree = 2
dp_degree = validate_3d_config(total_gpus, tp_degree, pp_degree)  # 128/(8*2) = 8 ✓
```

**原因**：
- DP 度 = 总 GPU 数 / (TP 度 × PP 度)
- DP 度必须 ≥ 1 且为整数
- DP 度为 0 → 无数据并行 → 全局批次=单卡批次 → 影响收敛

### 4. 错误：忽略网络拓扑，跨节点 TP 性能极差

**错误做法**：
```python
# 错误：TP 跨节点，网络带宽不足
# 节点内：NVLink 600GB/s
# 节点间：以太网 100Gbps（12.5GB/s）
# 性能差距：600GB/s vs 12.5GB/s = 48 倍

# 错误配置：8 卡 TP，分布在 2 个节点
tp_degree = 8
# 节点 0: GPU 0-3
# 节点 1: GPU 4-7
# ❌ 跨节点 TP 通信极慢
```

**正确做法**：
```python
# 正确：TP 限制在节点内，PP 跨节点
tp_degree = 4  # 限制在单节点内（A100 节点通常 4-8 卡）
pp_degree = 2  # 跨节点流水并行

# 节点 0: GPU 0-3 做 TP，持有 Layer 0-47
# 节点 1: GPU 4-7 做 TP，持有 Layer 48-95

# 或使用拓扑感知的调度
def auto_detect_topology():
    # 检测 NVLink 拓扑
    nvlink_groups = get_nvlink_groups()  # 如 [[0,1,2,3], [4,5,6,7]]
    
    # TP 度 ≤ NVLink 组大小
    tp_degree = min(desired_tp, len(nvlink_groups[0]))
    
    # PP 跨 NVLink 组
    pp_degree = len(nvlink_groups)
    
    return tp_degree, pp_degree
```

**原因**：
- TP 需要频繁 all-reduce（每个 Transformer 层）
- 节点间带宽远低于节点内（NVLink 600GB/s vs 以太网 12.5GB/s）
- 跨节点 TP → 通信成为瓶颈 → 性能下降 10 倍以上

### 5. 错误：DDP 未设置 `find_unused_parameters=True` 但存在未使用参数

**错误做法**：
```python
# 错误：模型有未使用参数，但未设置 find_unused_parameters
class MyModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.used_layer = nn.Linear(1024, 1024)
        self.unused_layer = nn.Linear(1024, 1024)  # ❌ 未在 forward 使用
    
    def forward(self, x):
        return self.used_layer(x)  # unused_layer 未使用

model = MyModel()
ddp_model = DDP(model, find_unused_parameters=False)  # ❌ 会报错
# RuntimeError: Expected to have finished reduction in the prior iteration before starting a new one.
```

**正确做法**：
```python
# 正确方案 1：设置 find_unused_parameters=True
ddp_model = DDP(model, find_unused_parameters=True)  # ✓ 但会增加开销

# 正确方案 2：删除未使用参数
class MyModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.used_layer = nn.Linear(1024, 1024)  # ✓ 只定义使用的层
    
    def forward(self, x):
        return self.used_layer(x)

# 正确方案 3：在 forward 中使用所有参数
class MyModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.layer1 = nn.Linear(1024, 1024)
        self.layer2 = nn.Linear(1024, 1024)
    
    def forward(self, x):
        x1 = self.layer1(x)
        x2 = self.layer2(x)
        return x1 + x2  # ✓ 使用所有参数
```

**原因**：
- DDP 默认假设所有参数都参与计算
- 未使用参数 → 梯度为 None → 梯度同步失败
- `find_unused_parameters=True` 会遍历计算图，增加 10-20% 开销

2. **Bubble 比例计算**：
```python
def calculate_bubble_ratio(pp_degree, num_micro_batches):
    """
    计算 PP 的 Bubble 比例
    Bubble 比例 = (PP度 - 1) / (微批次数 + PP度 - 1)
    """
    bubble_ratio = (pp_degree - 1) / (num_micro_batches + pp_degree - 1)
    return bubble_ratio

# 示例
print(calculate_bubble_ratio(4, 8))   # 0.27 (27%)
print(calculate_bubble_ratio(4, 16))  # 0.16 (16%)
```

3. **TP 切分矩阵乘法**：
```python
import torch.distributed as dist

def column_parallel_linear(x, weight, rank, world_size):
    """
    列并行线性层
    x: [batch, seq, in_features]
    weight: [out_features_per_partition, in_features]
    """
    # 本地计算
    output_local = x @ weight.t()  # [batch, seq, out_features_per_partition]
    
    # All-Reduce 汇聚
    output_global = torch.zeros_like(output_local)
    dist.all_reduce(output_local, op=dist.ReduceOp.SUM)
    
    return output_local  # 现在每个 GPU 都有完整输出
```

4. **3D 并行配置验证**：
```python
def validate_3d_config(total_gpus, tp_degree, pp_degree, dp_degree):
    """
    验证 3D 并行配置是否合法
    """
    # 检查 GPU 数量
    if total_gpus != tp_degree * pp_degree * dp_degree:
        raise ValueError(f"GPU 数量不匹配：{total_gpus} != {tp_degree}×{pp_degree}×{dp_degree}")
    
    # 检查 TP 度合理性
    if tp_degree > 8:
        print(f"警告：TP 度 {tp_degree} > 8，通信开销可能过大")
    
    # 检查 DP 度
    if dp_degree < 1:
        raise ValueError(f"DP 度 {dp_degree} < 1，无法进行数据并行")
    
    # 计算 Bubble 比例
    num_micro_batches = 16  # 假设
    bubble = (pp_degree - 1) / (num_micro_batches + pp_degree - 1)
    print(f"Bubble 比例：{bubble:.2%}")
    
    return True
```

# 生成模型评估：自动指标、人评与挑战

## 一句话结论
生成模型评估需要"自动指标+人工评测"双重体系：自动指标（BLEU/ROUGE/BERTScore）快但与人类偏好相关性低，人评（pairwise/rubric）准但成本高、一致性难保证；核心挑战在于"开放性答案无唯一标准"和"创造力与事实性的矛盾"。

## 核心定义/公式

### 1. 自动评估指标

#### BLEU（机器翻译）
```python
# BLEU-n: n-gram 精度 + 短句惩罚
BLEU = BP × exp(∑_{n=1}^N w_n log p_n)

# 其中:
# p_n = matched_ngram_count / candidate_ngram_count
# BP (brevity penalty) = exp(1 - r/c) if c < r else 1
# r = 参考文本长度, c = 候选文本长度

# 缺点: 只看精确匹配，忽略语义等价；对生成式任务不友好
```

#### ROUGE（文本摘要）
```python
# ROUGE-N: n-gram 召回率
ROUGE-N = ∑(matched_ngrams) / ∑(reference_ngrams)

# ROUGE-L: 最长公共子序列
ROUGE-L = (1 + β²) × Precision_L × Recall_L / (Recall_L + β² × Precision_L)

# β 控制召回率权重（通常 β 较大，强调召回）

# 缺点: 偏好字面匹配，忽略语义；同义改写得分低
```

#### BERTScore（语义相似度）
```python
# 基于 BERT embedding 的 token 级匹配
# 候选句子 C = [c1, c2, ..., cm]
# 参考句子 R = [r1, r2, ..., rn]

# Token 级相似度矩阵
S = cos_sim(embed(C), embed(R))  # [m, n]

# Recall: 每个参考 token 的最佳匹配
R_BERT = (1/|R|) ∑_j max_i S[i,j]

# Precision: 每个候选 token 的最佳匹配
P_BERT = (1/|C|) ∑_i max_j S[i,j]

# F1
F_BERT = 2 × P_BERT × R_BERT / (P_BERT + R_BERT)

# 优点: 捕捉语义相似性；同义替换也能得分
# 缺点: 计算成本高；依赖预训练模型质量
```

#### 事实性评估

**QA-based 方法**：
```python
# 流程:
# 1. 从生成文本中抽取事实陈述 (fact extraction)
# 2. 对每个事实生成问题 (question generation)
# 3. 用参考文档回答问题 (QA system)
# 4. 检查答案一致性 (answer consistency)

# 示例:
# 生成文本: "北京是中国的首都，人口超过3000万。"
# 抽取事实: "北京是中国的首都", "北京人口超过3000万"
# 生成问题: "中国的首都是哪里?", "北京的人口是多少?"
# 参考文档答案 vs 生成文本答案一致性

# 代表工具: FActScore, FACTSCORE, SAFE
```

**Entailment-based 方法**：
```python
# 使用 NLI 模型判断生成文本是否被参考文本蕴含
# Input: (premise=reference, hypothesis=generated)
# Output: entailment / contradiction / neutral

# 公式化:
P(entailment) = NLI_model(reference, generated)

# 优点: 直接度量事实一致性
# 缺点: NLI 模型本身的偏见；长文本需要切分
```

### 2. 人工评测方法

#### Pairwise Comparison
```python
# A/B 测试形式
# 给定: prompt, response_A, response_B
# 标注者选择: A更好 / B更好 / 差不多 / 都不好

# Elo Rating 计算:
# R_A' = R_A + K × (S_A - E_A)
# 其中:
#   S_A = 实际得分 (1胜/0负/0.5平)
#   E_A = 期望得分 = 1 / (1 + 10^((R_B-R_A)/400))
#   K = 学习率 (通常32)

# Chatbot Arena 使用的评分方式
```

#### Rubric-based Evaluation
```python
# 多维度打分 (1-5分或1-7分 Likert Scale)

# 常见维度:
rubric = {
    "helpfulness": "回复是否有帮助，是否解决用户问题",
    "relevance": "回复是否与问题相关，是否跑题",
    "accuracy": "事实是否准确，是否有幻觉",
    "coherence": "逻辑是否连贯，表达是否清晰",
    "safety": "是否有有害内容、偏见、歧视",
    "creativity": "是否有新颖性、创意性",
}

# 关键: 定义清晰的评分标准
# 示例 (helpfulness 5分):
# 5分: 完全解决问题，提供额外有价值信息
# 3分: 部分解决，有改进空间
# 1分: 没有帮助或误导
```

### 3. 创造性评估指标

```python
# 多样性指标
# Distinct-n: 唯一 n-gram 比例
Distinct_n = unique_ngrams / total_ngrams

# Self-BLEU: 生成样本之间的相似度（越低越多样）
# 用一个生成样本做 reference，其他做 candidate
Self_BLEU = average(BLEU(sample_i, samples_{-i}))

# Embedding-based Diversity
# 生成样本 embedding 的平均余弦距离
Diversity = 1 - mean(cos_sim(embed(s_i), embed(s_j)))

# 新颖度 (Novelty)
# 与训练集的差异程度
Novelty = 1 - max(similarity(generated, training_samples))
```

## 为什么（3 个因果链）

### 1. 为什么自动指标与人类评价相关性低？

```
开放性生成 → 无唯一标准答案 → 指标设计困难
     ↓
字面匹配指标(BLEU/ROUGE) → 同义改写得低分 → 与人类判断不一致
     ↓
人类判断标准复杂 → 涉及语义、流畅性、创造力、事实性 → 单一指标无法捕捉
```

**具体表现**：
- BLEU 对"the cat sat on the mat"和"the mat was sat on by the cat"评分相同（n-gram 一致），但人类可能认为前者更自然
- 模型可能生成高质量但字面不同的翻译（BLEU 低但人评高）
- 反之，模型可能 BLEU 高但语义错误（机械匹配）

### 2. 为什么人评成本高且一致性难保证？

```
人工标注需要专业知识 → 标注员培训成本高 → 规模化困难
           ↓
主观判断标准不统一 → 标注员间一致性低 → 需要多标注取平均
           ↓
标注疲劳和顺序效应 → 前后标准不一致 → 引入额外噪声
```

**量化证据**：
- 专业领域（代码、医疗）需要领域专家，时薪可达 $50-100
- 即使是通用对话，标注员间一致性（Kappa系数）通常只有 0.4-0.6
- 每个样本需要 3-5 个标注员才能达到可接受的信度

### 3. 为什么生成评估比分类评估更难？

```
分类任务 → 有明确标签 → 直接计算 acc/F1 → 指标可靠
     ↓
生成任务 → 开放性答案 → 无唯一正确解 → 指标设计难
     ↓
多维度权衡 → 准确性vs创造性vs安全性 → 单一指标无法综合
```

**具体差异**：
| 维度 | 分类评估 | 生成评估 |
|------|----------|----------|
| 答案空间 | 离散有限 | 开放无限 |
| 标准答案 | 存在 | 不存在或多元 |
| 评估维度 | 单一（对/错） | 多维（相关性、流畅性、事实性...） |
| 指标可靠性 | 高 | 低（与人类相关性通常 <0.5） |
| 评估成本 | 低 | 高 |

```python
from scipy.stats import spearmanr, pearsonr

def validate_metrics_against_human(auto_scores, human_scores):
    """验证自动指标与人类评分的相关性"""
    correlations = {}
    
    for metric_name, scores in auto_scores.items():
        spearman_corr, _ = spearmanr(scores, human_scores)
        pearson_corr, _ = pearsonr(scores, human_scores)
        correlations[metric_name] = {
            "spearman": spearman_corr,
            "pearson": pearson_corr
        }
    
    # 筛选与人类评分高相关的指标
    valid_metrics = [
        name for name, corr in correlations.items() 
        if abs(corr["spearman"]) > 0.3  # 经验阈值
    ]
    
    return correlations, valid_metrics
```

```python
            }
        }
```

```python
            metrics["factuality"] = self._init_factuality_checker()
        
        return metrics
    
    def evaluate_batch(
        self, 
        predictions: List[str], 
        references: List[str],
        source_docs: List[str] = None
    ) -> EvaluationResult:
        """批量评估"""
        
        # 1. 字面匹配指标
        bleu_scores = [self._compute_bleu(p, r) for p, r in zip(predictions, references)]
        rouge_scores = [self._compute_rouge(p, r) for p, r in zip(predictions, references)]

            predictions=predictions,
            references=references,

            lang="zh"
        )

        factuality_scores = None
        if source_docs:
            factuality_scores = [
                self._check_factuality(p, s) 
                for p, s in zip(predictions, source_docs)
            ]
        
        # 4. 多样性
        diversity_scores = self._compute_diversity(predictions)

            "bleu": np.mean(bleu_scores),
            "rouge_l": np.mean([r["rouge-l"] for r in rouge_scores]),
            "bertscore_f1": np.mean(bertscore_result["f1"]),
```

## 权衡分析

| 方案 | 收益 | 代价 | 适用边界 |
|------|------|------|----------|
| **BLEU/ROUGE** | 快速、无成本、可解释 | 与人类相关性低（<0.4）、忽略语义、偏好字面匹配 | 快速筛选、监控训练、机器翻译（补充） |
| **BERTScore** | 捕捉语义相似度、同义改写友好 | 计算成本高、依赖预训练模型、长文本需分段 | 语义理解任务、摘要、对话 |
| **QA-based 事实性** | 精确检测幻觉、可追溯错误来源 | 需要QA系统、依赖参考文档、对错误QA敏感 | 有参考文档的场景（摘要、RAG） |
| **NLI-based 事实性** | 无需QA系统、端到端评估 | 长文本切分损失上下文、NLI模型偏见 | 无参考文档场景、快速事实性筛查 |
| **Pairwise 人评** | 最接近真实偏好、相对判断更稳定 | 样本数 O(n²)、成本高、position bias | 模型对比、排行榜、关键决策 |
| **Rubric 人评** | 绝对评分、维度清晰、可诊断 | 一致性难保证、标注员培训成本 | 模型迭代、问题定位、多维度分析 |
| **LLM-as-Judge** | 成本低、一致性好、可规模化 | 模型偏见、self-preference、与真实人类有gap | 大规模初筛、自动排行榜、快速迭代 |

### 关键权衡决策树

```
需要评估生成模型
    │
    ├─ 有参考答案？
    │   ├─ 是 → BLEU/ROUGE 快速筛选 + BERTScore 语义评估
    │   └─ 否 → 跳过字面匹配，直接语义/质量评估
    │
    ├─ 有参考文档？
    │   ├─ 是 → 必须加事实性评估（QA-based 或 NLI-based）
    │   └─ 否 → 事实性评估可选
    │
    ├─ 评估预算？
    │   ├─ 高 → 人评（pairwise + rubric）
    │   ├─ 中 → LLM-as-Judge + 小规模人评校准
    │   └─ 低 → LLM-as-Judge + 自动指标
    │
    └─ 任务类型？
        ├─ 代码 → pass@k（执行正确率）
        ├─ 摘要 → ROUGE + BERTScore + 事实性
        ├─ 对话 → 事实性 + 安全性 + 人评
        └─ 创意 → 人评为主（多样性指标参考）
```

## 高频追问（8 个）

### 1. Q: BLEU 和 ROUGE 的区别是什么？分别适合什么场景？
A: BLEU 侧重**精度**（生成文本中有多少 n-gram 出现在参考文本中），适合机器翻译；ROUGE 侧重**召回**（参考文本中有多少 n-gram 被生成文本覆盖），适合文本摘要。BLEU 惩罚过短输出（brevity penalty），ROUGE 更宽松。

### 2. Q: 自动指标与人类评价相关性低，如何提高相关性？
A: 
1. **指标组合**：单一指标不足，组合多种指标（字面+语义+任务特定）
2. **任务定制**：根据任务特点调整指标权重（如对话重视事实性，创意重视多样性）
3. **校准**：在人评样本上拟合指标权重或训练回归模型
4. **使用更先进指标**：如 COMET（翻译）、BARTScore、GPTScore

### 3. Q: BERTScore 相比 BLEU/ROUGE 有什么优势？有什么局限？
A: 
**优势**：
- 捕捉语义相似度，同义词替换也能得分
- 对字面不同但语义相同的输出更友好
- 与人类相关性更高（通常 0.5-0.6 vs BLEU 的 0.3-0.4）

**局限**：
- 计算成本高（需要 BERT 前向传播）
- 依赖预训练模型质量，对领域外数据效果差
- 长文本需要分段处理，可能损失全局语义
- 对语言敏感（需要对应语言的 BERT 模型）

### 4. Q: 如何评估生成模型的幻觉问题？
A: 
1. **有参考文档场景**：
   - QA-based：生成问题，用参考文档回答，检查一致性
   - NLI-based：判断生成内容是否被参考文档蕴含
   
2. **无参考文档场景**：
   - 知识库检索：生成内容与知识库比对
   - 多模型投票：多个模型答案一致性
   - 置信度校准：模型不确定性作为幻觉信号
   
3. **人评**：让标注员检查事实准确性

### 5. Q: 人工评测中如何控制标注质量？
A: 
1. **标注员筛选**：
   - 资格测试（golden samples）
   - 语言能力测试（非母语者需额外培训）
   
2. **一致性监控**：
   - Fleiss' Kappa / Cohen's Kappa
   - 重复样本检查（5-10%样本双盲标注）
   
3. **质量控制机制**：
   - Golden samples 插入（检查标注员准确性）
   - 不合格标注重新分配
   - 标注员培训与反馈循环
   
4. **标注设计**：
   - 清晰的 rubric 和示例
   - 限制单个标注员时长（避免疲劳）
   - 随机打乱顺序（避免顺序效应）

### 6. Q: LLM-as-Judge 可信吗？有什么问题？
A: 
**可用场景**：
- 大规模初筛（快速迭代）
- 自动排行榜（成本可控）
- 问题模式识别（系统性错误）

**主要问题**：
1. **Self-preference**：模型倾向偏好自己风格
2. **Position bias**：A/B测试中对第一个或第二个有偏好
3. **Length bias**：偏好更长回复
4. **模型偏见**：对某些主题/风格有系统性偏好
5. **与人类 gap**：约 70-80% 一致率（视任务而定）

**缓解方法**：
- 交换顺序（position bias）
- 用多个 judge 模型投票
- 人工校准（人评样本调整 prompt）

### 7. Q: 如何评估模型的创造性？
A: 
**自动指标**（参考价值有限）：
- 多样性：Distinct-n、Self-BLEU
- 新颖度：与训练集/知识库的差异
- 惊喜度：生成概率分布的熵

**人工评估**（主要方法）：
- 新颖性维度：是否提供新观点/新角度
- 原创性：是否抄袭或模板化
- 有用创造性：是否在解决问题的前提下创新

**注意**：创造性与事实性可能冲突，需要单独维度评估

### 8. Q: Benchmark 是否能真实反映模型实际能力？
A: 
**不能完全反映，原因**：
1. **数据泄露**：测试集可能出现在训练数据中
2. **覆盖有限**：无法覆盖所有真实场景
3. **静态性**：真实世界动态变化，benchmark 静态
4. **过度优化**：模型针对 benchmark 优化
5. **简单模式**：benchmark 可能存在可利用的简单模式

**解决思路**：
1. **内部评测集**：构建覆盖真实业务场景的私有评测集
2. **动态更新**：定期更新 benchmark，避免泄露
3. **多维度评估**：结合 benchmark、人评、线上 A/B
4. **对抗样本**：构造困难样本和边缘情况
5. **A/B 测试**：最终以线上真实效果为准

## 常见错误（5 个）

### 1. 错误：只用 BLEU/ROUGE 评估生成模型
**描述**：认为 BLEU/ROUGE 足够，忽略语义相似度和人类偏好

**正确做法**：
- BLEU/ROUGE 用于快速筛选和监控，不是最终标准
- 必须补充语义指标（BERTScore）和事实性评估
- 关键决策需要人评确认

### 2. 错误：人评没有控制一致性和标注员质量
**描述**：直接发放标注任务，不检查标注员资质和一致性

**正确做法**：
- 标注前进行资格测试
- 计算 inter-annotator agreement（Fleiss' Kappa > 0.6）
- 插入 golden samples 监控标注质量
- 定期反馈和重新培训

### 3. 错误：评估维度不清晰或维度过多
**描述**：让标注员"整体打分"或定义 10+ 个维度，导致标准混乱

**正确做法**：
- 明确 3-5 个核心维度
- 每个维度有清晰的 rubric 和示例
- 避免维度重叠（如"准确性"和"事实性"）
- 先小规模测试 rubric 可理解性

### 4. 错误：忽略评估集与真实场景的 gap
**描述**：在干净评测集上效果好，线上效果差

**正确做法**：
- 评测集包含长尾、对抗、噪声样本
- 评测集分布接近真实业务分布
- 定期从线上采样补充评测集
- 线上 A/B 作为最终验证

### 5. 错误：评估结果不可复现
**描述**：评估代码、数据、prompt 未固定，结果无法复现

**正确做法**：
- 固定随机种子
- 版本化评测集和评估代码
- 记录所有配置（模型版本、prompt、采样参数）
- 保存中间结果便于审计

```python
# 自测题示例：Distinct-n 多样性计算
def compute_distinct_n(texts, n=2):
    """计算 Distinct-n 指标"""
    from collections import Counter
    
    all_ngrams = []
    for text in texts:
        tokens = text.split()
        ngrams = list(zip(*[tokens[i:] for i in range(n)]))
        all_ngrams.extend(ngrams)
    
    unique_ngrams = set(all_ngrams)
    distinct_ratio = len(unique_ngrams) / len(all_ngrams) if all_ngrams else 0
    
    return distinct_ratio
```

# 内部评测体系设计

## 一句话结论
内部评测体系 = 离线测试集（业务意图+长尾+对抗样本）+ 在线 A/B 测试 + 护栏指标监控，需覆盖模型全生命周期，平衡覆盖度、成本和反馈速度。

## 核心定义/公式

### 评测体系三层架构
```
离线评测 → 在线评测 → 长期监控
   ↓          ↓          ↓
快速迭代    真实效果    安全护栏
```

### 核心指标计算

#### 1. 离线评测指标
```python
# 综合得分计算
Score = α × Intent_Coverage + β × Longtail_Performance + γ × Adversarial_Robustness

其中：
- Intent_Coverage = 已测试意图数 / 总业务意图数
- Longtail_Performance = Σ(w_i × metric_i) / Σw_i  # 低频场景加权
- Adversarial_Robustness = 1 - Attack_Success_Rate
```

#### 2. 在线 A/B 测试指标
```python
# 主要业务指标
Response_Quality = Relevance × Accuracy × Helpfulness

# 护栏指标
Safety_Score = 1 - (Reject_Rate + Hallucination_Rate + Complaint_Rate) / 3

# 成本指标
Cost_Efficiency = Quality_Gain / (Latency × Token_Cost)
```

#### 3. 统计显著性计算
```python
import numpy as np
from scipy import stats

def compute_ab_significance(control_metrics, treatment_metrics, alpha=0.05):
    """
    A/B 测试显著性检验
    """
    # t-test for continuous metrics
    t_stat, p_value = stats.ttest_ind(control_metrics, treatment_metrics)
    
    # Cohen's d for effect size
    pooled_std = np.sqrt(
        (np.std(control_metrics)**2 + np.std(treatment_metrics)**2) / 2
    )
    cohens_d = (np.mean(treatment_metrics) - np.mean(control_metrics)) / pooled_std
    
    return {
        'p_value': p_value,
        'is_significant': p_value < alpha,
        'effect_size': cohens_d,
        'confidence_interval': stats.t.interval(
            0.95, 
            len(control_metrics) + len(treatment_metrics) - 2,
            loc=np.mean(treatment_metrics) - np.mean(control_metrics),
            scale=stats.sem(treatment_metrics - control_metrics)
        )
    }
```

## 为什么（2-3 个因果链）

### 1. 离线评测的必要性
**因果链**：模型迭代速度快 → 需要快速反馈机制 → 离线评测可低成本高频执行 → 但缺少真实用户交互 → 需要与在线评测结合

**关键原因**：
- **迭代效率**：离线评测可在分钟级完成，在线 A/B 需要天级甚至周级
- **安全性**：对抗样本必须先在离线环境测试，避免线上事故
- **可复现性**：离线测试集固定，便于追踪性能退化

### 2. 在线评测不可替代
**因果链**：真实用户行为复杂 → 离线评测覆盖有限 → 必须通过 A/B 验证实际效果 → 才能发现真实问题（如幻觉、拒答）

**关键原因**：
- **分布偏移**：用户实际输入与测试集分布不同
- **交互复杂性**：多轮对话、上下文依赖难以离线模拟
- **业务指标**：用户满意度、投诉率等只能在线收集

### 3. 护栏指标保障安全
**因果链**：模型能力提升 → 可能增加风险（幻觉、有害输出）→ 需要护栏指标持续监控 → 及时发现问题并回滚

**关键原因**：
- **幻觉率**：大模型倾向生成看似合理但错误的内容
- **拒答率**：过度拒答影响用户体验，拒答不足带来安全风险
- **投诉率**：用户直接反馈，最真实的效果信号

```python
# 在线A/B测试框架
class OnlineABFramework:
    def __init__(self, model_a, model_b, traffic_ratio=0.1):
        self.control = model_a      # 当前线上模型
        self.treatment = model_b    # 新模型
        self.traffic_ratio = traffic_ratio
        
    def design_experiment(self):
        """
        实验设计
        """
        return {
            'sample_size': self.compute_sample_size(
                baseline_metric=0.75,
                min_detectable_effect=0.02,
                power=0.8,
                alpha=0.05
            ),
            'randomization': 'user_id',  # 按用户分流
            'duration_days': 7,
            'guardrails': {
                'max_complaint_rate': 0.01,
                'max_latency_p99': 2000,  # ms
                'max_error_rate': 0.001
            }
        }
    
    def compute_sample_size(self, baseline_metric, min_detectable_effect, power, alpha):
        """
        计算所需样本量
        """
        from scipy import stats
        
        z_alpha = stats.norm.ppf(1 - alpha/2)
        z_beta = stats.norm.ppf(power)
        
        # Two-proportion test sample size
        p1 = baseline_metric
        p2 = baseline_metric + min_detectable_effect
        
        n = (
            (z_alpha * np.sqrt(2 * p1 * (1 - p1)) + 
             z_beta * np.sqrt(p1 * (1 - p1) + p2 * (1 - p2))) ** 2
        ) / ((p2 - p1) ** 2)
        
        return int(n * 2)  # 双组
    
    def collect_metrics(self, user_id, query, response, model_version):
        """
        收集在线指标
        """
        return {
            'user_id': user_id,
            'model_version': model_version,
            'query': query,
            'response': response,
            
            # 质量指标
            'response_time_ms': self.measure_latency(),
            'response_length': len(response),
            
            # 用户行为
            'user_feedback': self.get_user_rating(),  # 点赞/点踩
            'regenerate': self.check_regeneration(),
            'copy_action': self.check_copy(),
            
            # 安全指标
            'is_rejected': self.detect_rejection(response),
            'hallucination_flag': self.detect_hallucination(response),
            
            # 成本指标
            'input_tokens': self.count_tokens(query),
            'output_tokens': self.count_tokens(response),
            'gpu_time_ms': self.measure_compute_time()
        }
    
    def analyze_results(self):
        """
        分析实验结果
        """
        control_data = self.load_experiment_data('control')
        treatment_data = self.load_experiment_data('treatment')
        
        results = {}
        
        # 主要指标
        for metric in ['user_satisfaction', 'response_quality', 'task_completion']:
            results[metric] = self.compute_ab_significance(
                control_data[metric],
                treatment_data[metric]
            )
        
        # 护栏指标
        for guardrail in ['complaint_rate', 'hallucination_rate', 'latency_p99']:
            control_value = np.mean(control_data[guardrail])
            treatment_value = np.mean(treatment_data[guardrail])
            
            # 检查是否超过阈值
            results[guardrail] = {
                'control': control_value,
                'treatment': treatment_value,
                'passes': treatment_value <= self.guardrails.get(f'max_{guardrail}', float('inf'))
            }
        
        return results
```

#### 步骤 3：设计护栏指标监控
```python
# 护栏指标监控系统
class GuardrailMonitor:
    def __init__(self):
        self.thresholds = {
            'reject_rate': {
                'warning': 0.05,
                'critical': 0.10,
                'description': '拒答率过高影响用户体验'
            },
            'hallucination_rate': {
                'warning': 0.03,
                'critical': 0.05,
                'description': '幻觉率过高导致用户误导'
            },
            'complaint_rate': {
                'warning': 0.01,
                'critical': 0.02,
                'description': '投诉率直接反映用户不满'
            },
            'latency_p99': {
                'warning': 2000,  # ms
                'critical': 3000,
                'description': '延迟过高影响体验'
            }
        }
    
    def detect_hallucination(self, response, context=None):
        """
        幻觉检测（多维度）
        """
        hallucination_score = 0.0
        
        # 方法1：基于事实核查
        if self.has_factual_claims(response):
            fact_check_result = self.fact_check_api(response)
            if not fact_check_result['is_factual']:
                hallucination_score += 0.5
        
        # 方法2：基于一致性检查
        if context:
            consistency = self.check_consistency(response, context)
            if consistency < 0.7:
                hallucination_score += 0.3
        
        # 方法3：基于模型self-evaluation
        self_eval = self.model_self_evaluate(response)
        if self_eval['confidence'] < 0.6:
            hallucination_score += 0.2
        
        return hallucination_score > 0.6
    
    def compute_guardrail_metrics(self, time_window='1h'):
        """
        计算护栏指标
        """
        metrics = {}
        
        # 拒答率
        metrics['reject_rate'] = self.count_rejections(time_window) / self.total_queries(time_window)
        
        # 幻觉率（抽样检测）
        sample_responses = self.sample_responses(time_window, sample_rate=0.1)
        hallucinations = [self.detect_hallucination(r) for r in sample_responses]
        metrics['hallucination_rate'] = sum(hallucinations) / len(hallucinations)
        
        # 投诉率
        metrics['complaint_rate'] = self.count_complaints(time_window) / self.total_queries(time_window)
        
        # 延迟P99
        latencies = self.get_latencies(time_window)
        metrics['latency_p99'] = np.percentile(latencies, 99)
        
        # 成本指标
        metrics['cost_per_query'] = self.compute_cost(time_window) / self.total_queries(time_window)
        
        return metrics
    
    def trigger_alerts(self, metrics):
        """
        触发告警
        """
        alerts = []
        
        for metric_name, value in metrics.items():
            if metric_name in self.thresholds:
                threshold = self.thresholds[metric_name]
                
                if value >= threshold['critical']:
                    alerts.append({
                        'level': 'CRITICAL',
                        'metric': metric_name,
                        'value': value,
                        'threshold': threshold['critical'],
                        'action': '立即回滚模型'
                    })
                elif value >= threshold['warning']:
                    alerts.append({
                        'level': 'WARNING',
                        'metric': metric_name,
                        'value': value,
                        'threshold': threshold['warning'],
                        'action': '加强监控，准备降级方案'
                    })
        
        return alerts
```

```python
# 评测流程编排
class EvaluationPipeline:
    def __init__(self, config):
        self.offline_eval = OfflineEvaluator(config['offline'])
        self.online_ab = OnlineABFramework(config['online'])
        self.guardrail_monitor = GuardrailMonitor()
        
    def run_full_evaluation(self, model_version):
        """
        完整评测流程
        """
        results = {
            'model_version': model_version,
            'timestamp': datetime.now()
        }
        
        # 阶段1：离线评测（必须通过才能上线）
        print("Stage 1: Offline Evaluation...")
        offline_results = self.offline_eval.evaluate(model_version)
        results['offline'] = offline_results
        
        if not self.passes_offline_gate(offline_results):
            print("❌ Offline evaluation failed. Stop deployment.")
            return results
        
        # 阶段2：小流量A/B测试
        print("Stage 2: Small-traffic A/B Test...")
        ab_results = self.online_ab.run_experiment(
            traffic_ratio=0.05,  # 5%小流量
            duration_days=3
        )
        results['online_ab'] = ab_results
        
        if not self.passes_ab_gate(ab_results):
            print("❌ A/B test failed. Rollback.")
            self.online_ab.rollback()
            return results
        
        # 阶段3：全量上线 + 持续监控
        print("Stage 3: Full Deployment with Monitoring...")
        self.online_ab.full_deployment()
        
        # 持续监控护栏指标
        monitoring_results = self.guardrail_monitor.continuous_monitor(
            interval_minutes=30,
            alert_callback=self.handle_alert
        )
        results['monitoring'] = monitoring_results
        
        return results
    
    def passes_offline_gate(self, results):
        """
        离线评测通过条件
        """
        return (
            results['intent_accuracy'] >= 0.85 and
            results['longtail_performance'] >= 0.75 and
            results['adversarial_robustness'] >= 0.90
        )
    
    def passes_ab_gate(self, results):
        """
        A/B测试通过条件
        """
        return (
            results['user_satisfaction']['is_significant'] and
            results['user_satisfaction']['effect_size'] > 0.02 and
            all(g['passes'] for g in results['guardrails'].values())
        )
    
    def handle_alert(self, alert):
        """
        处理告警
        """
        if alert['level'] == 'CRITICAL':
            # 自动回滚
            self.online_ab.rollback()
            self.notify_team(alert)
        elif alert['level'] == 'WARNING':
            # 降低流量
            self.online_ab.reduce_traffic(ratio=0.5)
            self.notify_team(alert)
```

## 权衡分析

| 方案 | 收益 | 代价 | 适用边界 |
|------|------|------|----------|
| **离线评测为主** | 快速迭代、低成本、可复现 | 缺少真实用户反馈、覆盖有限 | 模型初期开发、高频迭代阶段 |
| **在线A/B为主** | 真实效果验证、用户行为数据 | 周期长、成本高、有风险 | 模型上线前验证、重大版本更新 |
| **全量上线监控** | 最大流量、最快发现问题 | 风险高、影响面大 | 仅限经过充分测试的稳定版本 |
| **小流量灰度** | 平衡风险和验证效果 | 样本量小、统计显著性不足 | 新功能上线、模型微调 |
| **自动化评测** | 效率高、可规模化 | 可能漏掉复杂问题 | 标准化场景、高频次评测 |
| **人工评测** | 发现深层次问题、质量高 | 成本高、速度慢 | 关键场景、对抗样本、质量把控 |

### 详细权衡案例

#### 案例1：测试集规模 vs 评测成本
```python
# 测试集规模权衡
def compute_optimal_test_size(available_budget, cost_per_sample):
    """
    计算最优测试集规模
    
    权衡因素：
    - 样本太少：覆盖不足，漏掉重要问题
    - 样本太多：成本高，边际收益递减
    """
    # 覆盖度曲线（对数增长）
    coverage = lambda n: 1 - np.exp(-n / 500)
    
    # 边际收益递减
    marginal_benefit = lambda n: coverage(n+1) - coverage(n)
    
    # 最优点：边际收益 = 边际成本
    # coverage'(n) = cost_per_sample / available_budget
    optimal_n = 500 * np.log(available_budget / (500 * cost_per_sample))
    
    return {
        'optimal_size': int(optimal_n),
        'expected_coverage': coverage(optimal_n),
        'total_cost': optimal_n * cost_per_sample,
        'roi': coverage(optimal_n) / (optimal_n * cost_per_sample / available_budget)
    }
```

```python
def compute_ab_duration(daily_traffic, expected_effect_size, power=0.8, alpha=0.05):
    """
    计算A/B测试所需时长
    
    权衡因素：
    - 时间短：样本不足，无法检测到效果
    - 时间长：延迟决策，错过机会窗口

            'longer_test': '决策延迟，成本增加',
            'optimal': f'{required_days:.1f}天，平衡显著性和效率'
```

## 高频追问（至少 5 个）

### 1. Q: Benchmark 是否能真实反映模型实际能力？
**A:** 不能完全反映，需要多维验证：
- **局限性**：Benchmark 是静态的、有限的，无法覆盖真实用户的复杂交互和长尾场景
- **分布偏移**：测试集分布与实际用户query分布不一致
- **解决方案**：
  - 动态更新测试集，加入真实用户query（脱敏后）
  - 离线+在线评测结合，用A/B验证真实效果
  - 监控护栏指标（投诉率、幻觉率）作为补充
  - 定期人工评测关键场景

### 2. Q: 如何平衡测试集的覆盖度和评测成本？
**A:** 采用分层采样 + 聚焦高价值场景：
- **分层策略**：核心意图（60%）+ 边缘意图（30%）+ 新增意图（10%）
- **重要性采样**：高频场景高权重，低频场景保证多样性
- **自动化生成**：用LLM生成测试样本，人工审核关键部分
- **增量更新**：定期补充新发现的bad case，淘汰过时样本
- **ROI导向**：计算每个测试样本的边际收益，优先投入高价值场景

### 3. Q: 对抗样本测试集如何构建？如何避免过拟合？
**A:** 多源构建 + 持续更新 + 保密性：
- **构建方法**：
  - 红队测试：安全团队人工构造攻击样本
  - 自动化生成：基于规则模板（越狱、注入、误导）
  - 真实案例：从用户投诉中提取
  - 对抗训练：用对抗模型生成困难样本
- **避免过拟合**：
  - 测试集保密，不用于训练
  - 定期更换测试集（每季度20%更新）
  - 保留holdout set，用于最终验证
  - 关注模型在未见对抗样本上的泛化能力

### 4. Q: 如何设计有效的在线A/B测试？需要考虑哪些因素？
**A:** 系统化设计 + 统计严谨 + 护栏机制：
- **实验设计**：
  - 随机分流：按user_id分流，保证组间可比性
  - 样本量计算：基于MDE（最小检测效应）和power
  - 实验时长：至少7天，覆盖用户行为周期
- **关键因素**：
  - 季节性：避免节假日等特殊时期
  - 新奇效应：用户对新功能的初始兴趣
  - 网络效应：用户间相互作用（社交场景）
- **护栏机制**：
  - 预设停止条件：投诉率、错误率阈值
  - 自动回滚：触发critical阈值立即回滚
  - 灰度发布：小流量→中流量→全量

### 5. Q: 护栏指标如何选择？阈值如何确定？
**A:** 业务导向 + 数据驱动 + 动态调整：
- **选择原则**：
  - 安全性：拒答率、有害内容率
  - 质量：幻觉率、投诉率
  - 性能：延迟P99、错误率
  - 成本：token消耗、GPU利用率
- **阈值确定**：
  - 历史数据：基于过去3个月的基线
  - 业务要求：根据SLA和用户容忍度
  - 竞品对标：参考行业标准
  - 分级设置：warning（预警）+ critical（熔断）
- **动态调整**：
  - 季度复盘：根据实际情况调整阈值
  - A/B对比：新模型阈值应不劣于基线

### 6. Q: 如何评估评测体系本身的有效性？
**A:** 元评测 + 反馈闭环：
- **有效性指标**：
  - 问题发现率：线上问题中有多少被离线评测发现
  - 误报率：离线测试问题中有多少线上未发生
  - 预测准确性：离线得分与线上效果的Correlation
- **反馈闭环**：
  - 线上问题 → 补充到测试集 → 下次评测发现
  - 定期审计：每月分析评测体系的coverage和recall
  - AB测试验证：离线评测好的模型，线上是否真的好

### 7. Q: 对于多语言、多模态模型，评测体系如何设计？
**A:** 分层评测 + 领域适配：
- **多语言**：
  - 核心语言：完整评测集（如中英文）
  - 其他语言：抽样评测 + 自动翻译校验
  - 语言特定测试：语法、文化适应性
- **多模态**：
  - 单模态基线：文本、图像分别评测
  - 跨模态任务：图文理解、图像生成质量
  - 一致性评测：多模态输出是否一致
- **资源分配**：
  - 核心场景：深度评测（人工+自动）
  - 长尾场景：自动化评测为主
  - 根据业务重要性分配资源

## 常见错误（至少 3 个）

### 1. 只依赖公开Benchmark，忽视业务特定评测
**错误做法**：
- 只在MMLU、GSM8K等公开数据集上评测
- 忽视用户真实场景和业务意图

**正确做法**：
```python
# 构建业务特定评测集
business_intents = [
    '查询订单状态',
    '退换货咨询',
    '产品推荐',
    '投诉处理',
    # ...
]

# 为每个意图构建测试集
for intent in business_intents:
    test_cases = collect_real_queries(intent, past_days=90)
    eval_set.extend(test_cases)

# 计算业务覆盖率
coverage = len(eval_set) / total_business_intents
```

**后果**：模型在公开benchmark表现好，但上线后业务指标未提升，甚至下降

### 2. A/B测试样本量不足，过早得出结论
**错误做法**：
- 只测试2-3天就做决策
- 流量太小（如1%），无法检测到小效应
- 忽视统计显著性，只看平均值差异

**正确做法**：
```python
# 正确的A/B测试流程
def proper_ab_test():
    # 1. 计算所需样本量
    required_n = compute_sample_size(
        baseline=0.75,
        mde=0.02,  # 希望检测到2%的提升
        power=0.8,
        alpha=0.05
    )
    
    # 2. 确定测试时长
    daily_traffic = 10000
    test_days = required_n / (daily_traffic * 0.1)  # 10%流量
    test_days = max(test_days, 7)  # 至少7天
    
    # 3. 检查显著性
    results = run_ab_test(duration_days=test_days)
    
    if not results['is_significant']:
        print("样本不足，无法得出结论。延长测试时间或增加流量。")
        return
    
    # 4. 检查效应量
    if results['effect_size'] < mde:
        print("提升不显著，不建议上线。")
        return
```

**后果**：基于不显著的结果做决策，可能引入无效果甚至负效果的模型

### 3. 忽视护栏指标，导致上线后出现安全问题
**错误做法**：
- 只关注业务指标（满意度、转化率）
- 忽视拒答率、幻觉率、投诉率等安全指标
- 没有预设熔断机制

**正确做法**：
```python
# 完整的上线checklist
def pre_launch_checklist(model_version):
    checks = {
        'business_metrics': {
            'user_satisfaction': evaluate_satisfaction(),
            'task_completion': evaluate_task_completion(),
            'response_quality': evaluate_quality()
        },
        'guardrails': {
            'reject_rate': evaluate_reject_rate(),
            'hallucination_rate': evaluate_hallucination(),
            'complaint_rate': evaluate_complaints(),
            'harmful_content_rate': evaluate_safety()
        },
        'performance': {
            'latency_p99': measure_latency(),
            'error_rate': measure_errors(),
            'cost_per_query': measure_cost()
        }
    }
    
    # 所有护栏指标必须通过
    for metric, value in checks['guardrails'].items():
        if value > GUARDRAIL_THRESHOLDS[metric]['critical']:
            raise Exception(f"护栏指标 {metric} 超过临界值: {value}")
    
    return checks
```

**后果**：模型上线后出现严重幻觉、有害内容等问题，被迫紧急回滚，影响用户体验和品牌形象

### 4. 测试集泄露到训练数据，导致评测失真
**错误做法**：
- 使用相同的数据训练和评测
- 测试集被标注人员泄露到训练集
- 数据增强时过拟合测试集特征

**正确做法**：
```python
# 严格的数据隔离
class DataIsolation:
    def __init__(self):
        self.train_data = set()
        self.eval_data = set()
    
    def add_to_train(self, samples):
        # 确保评测数据不在训练集中
        for sample in samples:
            sample_hash = self.hash_sample(sample)
            if sample_hash in self.eval_data:
                raise ValueError("数据泄露：评测样本进入训练集")
            self.train_data.add(sample_hash)
    
    def add_to_eval(self, samples):
        # 确保训练数据不在评测集中
        for sample in samples:
            sample_hash = self.hash_sample(sample)
            if sample_hash in self.train_data:
                raise ValueError("数据泄露：训练样本进入评测集")
            self.eval_data.add(sample_hash)
    
    def hash_sample(self, sample):
        # 基于内容hash，避免表面不同但语义相同的样本
        return hash(normalize_text(sample['query']))
```

**后果**：评测分数虚高，但线上效果差，无法发现模型的真实问题

### 5. 长尾场景评测覆盖不足
**错误做法**：
- 只测试高频场景（80%用户query）
- 忽视低频但重要的长尾场景
- 没有专门的longtail测试集

**正确做法**：
```python
# 长尾场景专门评测
def evaluate_longtail(model, test_set):
    # 按频率分组
    freq_groups = {
        'high_freq': filter_by_frequency(test_set, min_freq=1000),
        'medium_freq': filter_by_frequency(test_set, min_freq=100, max_freq=1000),
        'low_freq': filter_by_frequency(test_set, max_freq=100)
    }
    
    # 分别评测
    results = {}
    for group_name, samples in freq_groups.items():
        results[group_name] = {
            'accuracy': evaluate_accuracy(model, samples),
            'coverage': len(samples) / len(test_set),
            'avg_performance': evaluate_performance(model, samples)
        }
    
    # 重点检查低频场景表现
    if results['low_freq']['accuracy'] < 0.6:
        print("⚠️  长尾场景表现不佳，需要改进")
    
    return results
```

**后果**：模型在高频场景表现好，但长尾场景差，导致部分用户体验极差，影响口碑

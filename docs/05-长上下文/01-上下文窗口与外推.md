# 上下文窗口与外推问题

## 一句话结论
训练侧通过继续预训练和长文本数据 packing 扩展窗口；推理侧用 RoPE scaling（线性/NTK/YaRN）调整位置编码频率，需权衡注意力模式改变、近邻偏置、显存峰值与精度损失。

## 核心定义/公式

### RoPE 外推基础
位置编码将绝对位置 $m$ 映射到旋转矩阵，嵌入到注意力计算中：

$$
\text{RoPE}(x_m, m) = 
\begin{pmatrix}
x_m^{(1)} \\
x_m^{(2)} \\
\vdots \\
x_m^{(d-1)} \\
x_m^{(d)}
\end{pmatrix}
\odot
\begin{pmatrix}
\cos(m\theta_1) \\
\sin(m\theta_1) \\
\vdots \\
\cos(m\theta_{d/2}) \\
\sin(m\theta_{d/2})
\end{pmatrix}
$$

其中 $\theta_i = 10000^{-2i/d}$ 为频率基。

### 外推扩展方法

#### 1. 线性插值（Linear Scaling）
直接缩放位置索引，将位置 $m$ 压缩到训练范围内：

$$
m' = \frac{m}{\alpha}, \quad \alpha = \frac{L_{\text{target}}}{L_{\text{train}}}
$$

#### 2. NTK-aware Scaling
调整频率基而非位置索引，保持高频分辨率：

$$
\theta'_i = \theta_i \cdot \alpha^{-2i/d}
$$

#### 3. YaRN（Yet another RoPE extensioN）
结合 NTK-aware scaling 与温度调整：

$$
\theta'_i = \theta_i \cdot \left(\frac{\alpha \cdot d}{d-2}\right)^{-2i/d}, \quad t = 0.07 \cdot \ln(\alpha) + 1.0
$$

注意力分数额外乘温度系数 $t$：

$$
\text{Attention} = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k} \cdot t}\right)V
$$

### 显存估算公式

KV cache 显存占用（单层）：

$$
\text{Memory}_{\text{KV}} = 2 \times n_{\text{layers}} \times n_{\text{heads}} \times d_{\text{head}} \times L \times \text{dtype\_bytes}
$$

对于 GQA：

$$
\text{Memory}_{\text{KV}}^{\text{GQA}} = 2 \times n_{\text{layers}} \times n_{\text{kv\_heads}} \times d_{\text{head}} \times L \times \text{dtype\_bytes}
$$

### 代码示例

```python
import torch
import math

def get_rope_scaling_factors(original_max_len, target_max_len, dim, method='yarn'):
    """
    计算 RoPE scaling 参数
    
    Args:
        original_max_len: 训练时的最大长度
        target_max_len: 目标扩展长度
        dim: 釖量维度
        method: 'linear' | 'ntk' | 'yarn'
    """
    alpha = target_max_len / original_max_len
    
    # 基础频率
    base_freq = 10000.0
    inv_freq = 1.0 / (base_freq ** (torch.arange(0, dim, 2).float() / dim))
    
    if method == 'linear':
        # 线性插值：只缩放位置索引
        scaling_factor = 1.0 / alpha
        return {'type': 'linear', 'factor': scaling_factor}
    
    elif method == 'ntk':
        # NTK-aware: 调整频率基
        ntk_alpha = alpha ** (dim / (dim - 2))
        scaled_inv_freq = inv_freq / ntk_alpha
        return {'type': 'ntk', 'inv_freq': scaled_inv_freq}
    
    elif method == 'yarn':
        # YaRN: NTK + 温度调整
        ntk_alpha = (alpha * dim / (dim - 2)) ** (dim / (dim - 2))
        scaled_inv_freq = inv_freq / ntk_alpha
        
        # 温度系数
        temp = 0.07 * math.log(alpha) + 1.0
        
        return {
            'type': 'yarn',
            'inv_freq': scaled_inv_freq,
            'temperature': temp
        }

# 使用示例（Transformers 库）
from transformers import AutoModelForCausalLM, AutoConfig

config = AutoConfig.from_pretrained("Qwen/Qwen-7B")
# 修改 RoPE scaling
config.rope_scaling = {
    "type": "yarn",
    "factor": 4.0,  # 4倍外推
}
model = AutoModelForCausalLM.from_pretrained("Qwen/Qwen-7B", config=config)
```

## 为什么（3 个因果链）

### 1. 短文本模型为何无法直接处理长文本

**现象**：模型在训练时只见过长度 $L_{\text{train}}$ 内的位置编码，推理时遇到 $L > L_{\text{train}}$ 的位置时性能急剧下降。

**根因**：
- **位置分布外推**：RoPE 位置编码使模型学到特定位置范围内的注意力模式。训练时 $m \in [0, L_{\text{train}})$，推理时 $m \geq L_{\text{train}}$ 属于 OOD（Out-of-Distribution）。
- **频率分辨率不足**：高频成分（小 $i$）的 $\cos(m\theta_i), \sin(m\theta_i)$ 在长位置下周期重复，丢失位置区分度。
- **注意力模式错配**：模型学到的相对位置偏置（如"距离 100 的 token 权重较低"）在长上下文中无法泛化。

**结果**：长文本困惑度飙升、关键信息检索失败、"大海捞针"任务准确率断崖下跌。

### 2. 扩展位置编码为何有副作用

**现象**：应用 RoPE scaling 后，虽然能处理更长序列，但短文本性能下降、长程依赖建模变弱。

**根因**：
- **注意力模式改变**：线性插值压缩位置，使"相对距离"被人为缩小。原本距离 100 的 token 现在距离被映射为 25，导致注意力分布过度集中在近邻（近邻偏置）。
- **频率失真**：NTK/YaRN 调整频率基后，不同维度的位置编码分辨能力不均。低频成分（大 $i$）外推能力强，但高频成分（小 $i$）可能退化。
- **训练-推理不一致**：推理时突然改变位置编码，模型未见过这种分布，需要重新适应。

**结果**：
- 短文本任务性能下降 5-10%（推理/摘要）
- 长程依赖检索准确率下降（距离越远越明显）
- 温度调整虽缓解但不完美，极端长距离（如 100K tokens）仍需继续预训练

### 3. 长上下文为何"显存炸"

**现象**：将上下文从 4K 扩展到 128K，显存占用增长 32 倍，单卡 A100 80GB 无法承载。

**根因**：
- **KV cache 线性增长**：KV cache 大小 $\propto L$（序列长度）$\times$ batch_size。128K 上下文 + batch=1 的 KV cache 可能超过 20GB。
- **注意力矩阵显存峰值**：虽然 FlashAttention 避免存储完整 $L \times L$ 矩阵，但 prefill 阶段仍需中间缓冲区。序列越长，峰值越高。
- **激活重算开销**：activation checkpointing 虽节省显存，但增加计算量。长序列下重算成本不可忽视。
- **碎片化**：动态 batch + 长 KV cache 导致显存碎片，实际占用高于理论值。

**结果**：
- 128K 上下文在 7B 模型上需 4×A100 80GB（TP=4）
- 吞吐下降（batch size 受限）
- 成本飙升（需要高端显卡或多卡推理）

## 怎么做（可落地步骤）

### 训练侧扩展上下文窗口

#### 方法 1：继续预训练（Continued Pretrain）

**步骤**：
1. **准备长文本数据**
   - 收集长度 > 目标窗口的文档（如书籍、代码库、长对话）
   - 数据质量筛选：去重、清洗、领域平衡
   - 目标数据量：至少 1B tokens（经验值）

2. **调整训练配置**
   ```python
   # 训练参数示例
   training_args = {
       "max_length": 32768,  # 目标窗口
       "learning_rate": 1e-5,  # 比预训练低
       "warmup_steps": 200,
       "num_train_epochs": 1,
       "packing": True,  # 关键：使用 packing 提升效率
   }
   ```

3. **Packing 策略**
   ```python
   # 短文档拼接成长序列
   def pack_documents(docs, max_length):
       packed = []
       current = []
       current_len = 0
       
       for doc in docs:
           if current_len + len(doc) <= max_length:
               current.append(doc)
               current_len += len(doc)
           else:
               packed.append(current)
               current = [doc]
               current_len = len(doc)
       
       return packed
   ```

4. **验证与评估**
   - 在长文本 benchmark 测试（如 LongBench、L-Eval）
   - 短文本任务回归测试（确保不遗忘）
   - "大海捞针"测试（关键信息检索）

**关键配置**：
- `packing=True`：提升数据效率，避免 padding 浪费
- `gradient_checkpointing=True`：节省激活显存
- `max_length`：逐步增加（如 4K → 16K → 32K），而非直接跳到 128K

#### 方法 2：长窗口微调

**步骤**：
1. 使用 RoPE scaling 配置加载模型
2. 在长文本 SFT 数据上微调 1-2 epoch
3. 学习率设为预训练的 10-20%

```python
from transformers import TrainingArguments
```

### 推理侧外推（无需训练）

#### 方案选择决策树

```
目标长度 vs 训练长度
├─ 2倍以内
│   └─ 线性插值（简单快速，副作用可控）
├─ 2-8倍
│   └─ NTK-aware scaling（平衡精度与外推能力）
└─ >8倍
    └─ YaRN（最先进，需调温度参数）
```

```python
# 滑动窗口配置（仅保留最近 W 个 token 的 KV）
class SlidingWindowCache:
    def __init__(self, window_size=4096):
        self.window_size = window_size
        self.cache = []
    
    def update(self, new_kv):
        self.cache.append(new_kv)
        if len(self.cache) > self.window_size:
            self.cache.pop(0)
        return self.cache
```

**适用场景**：
- 文档问答（只需关注局部上下文）
- 代码补全（局部依赖）
- 不适合长程依赖任务

```python
def benchmark_long_context(model, tokenizer, lengths=[4096, 8192, 16384, 32768]):
    """
    评估不同长度下的性能
    """
    import time
    
    results = []
    for length in lengths:
        # 构造测试文本
        text = "测试文本。" * (length // 5)

        torch.cuda.empty_cache()
        torch.cuda.reset_peak_memory_stats()
        
        start = time.time()
        with torch.no_grad():
            outputs = model(**inputs)
        elapsed = time.time() - start
        
        peak_memory = torch.cuda.max_memory_allocated() / 1e9  # GB
        
        results.append({
            "length": length,
            "time": elapsed,
            "peak_memory": peak_memory,
        })
    
    return results
```

## 权衡分析

| 方案 | 收益 | 代价 | 适用边界 |
|------|------|------|----------|
| **线性插值** | 实现简单，无需训练；2倍外推性能稳定 | 近邻偏置严重；长程依赖能力下降 | 外推 < 2倍；对精度要求不高 |
| **NTK-aware** | 平衡短/长文本性能；外推能力优于线性 | 需调参；极端长度（>8倍）仍不稳定 | 外推 2-8倍；中长上下文场景 |
| **YaRN** | 外推能力强（可到 16倍）；副作用最小 | 实现复杂；温度参数需调优 | 外推 > 8倍；高质量长文本任务 |
| **继续预训练** | 最优长文本能力；可控质量 | 成本高（数据+算力）；周期长 | 产品级长上下文；预算充足 |
| **滑动窗口** | 显存占用固定；适合极长文本（>100K） | 长程依赖完全丢失；不适合检索 | 流式处理；代码补全；对话历史 |
| **KV cache 量化（INT8）** | 显存减半；吞吐提升 | 长文本精度损失明显 | 显存紧张；可容忍 3-5% 性能下降 |
| **GQA 架构** | KV cache 减少 75%；推理加速 | 模型能力略降（vs MHA） | 推理优先；长上下文场景 |
| **FlashAttention** | 显存大幅降低；长序列加速 | 需特定 GPU（A100/A10 等） | 几乎所有长上下文场景 |

**关键权衡点**：
1. **外推倍数 vs 精度损失**：外推越大，短文本能力下降越明显。需在业务场景中测试权衡。
2. **显存 vs 吞吐**：KV cache 量化、GQA 节省显存，但可能降低 batch size 上限，需实测吞吐。
3. **训练成本 vs 推理质量**：继续预训练成本高但质量好；推理侧外推成本低但有上限。

## 高频追问（8 个）

### 1. Q: 训练侧扩展上下文窗口，数据量需要多少？

A: **经验法则**：目标窗口 × 1000 个样本。例如扩展到 32K，需准备至少 32M tokens 长文本数据。数据质量比数量更重要：
- 去重率 > 95%
- 领域覆盖全面（代码/书籍/对话/文档）
- 长度分布：至少 50% 样本 > 目标窗口的 50%

数据不足时，可用 packing 拼接短文本，但效果不如真实长文本。

### 2. Q: 推理侧外推方法如何选择？

A: **决策依据**：
1. **外推倍数**：
   - < 2倍：线性插值
   - 2-8倍：NTK-aware
   - > 8倍：YaRN
2. **业务容忍度**：
   - 短文本任务多：优先 NTK/YaRN（保护短文本能力）
   - 只做长文本：线性插值足够
3. **工程成本**：
   - 快速上线：线性插值（改动最小）
   - 产品级：YaRN（需验证效果）

**实测建议**：用"大海捞针"测试 + LongBench benchmark 验证后再上线。

### 3. Q: 扩展位置编码后，短文本性能下降多少？

A: **典型数据**（基于 Qwen-7B，4K → 32K 外推）：
- 线性插值：短文本任务下降 8-12%
- NTK-aware：下降 5-8%
- YaRN：下降 3-5%

**缓解方法**：
1. 温度调整（YaRN 自动处理）
2. 混合训练：用 10% 短文本数据混合长文本继续预训练
3. 多模型策略：短文本用原模型，长文本用外推模型

### 4. Q: 显存优化方法的优先级如何排序？

A: **优先级排序**（性价比从高到低）：
1. **FlashAttention**：几乎无代价，显存降低 50%+
2. **GQA 模型选择**：选 Qwen/LLaMA-3 等原生支持 GQA 的模型
3. **KV cache 量化**：INT8 减半显存，精度损失可控
4. **PagedAttention（vLLM）**：提升吞吐，避免碎片
5. **滑动窗口**：适合特定场景，长程依赖丢失
6. **模型并行（TP/PP）**：显存不足时的最后手段

**组合策略**：FlashAttention + GQA + INT8 KV cache 可在单卡 A100 上跑 32K 上下文（7B 模型）。

### 5. Q: 哪些场景不适合直接外推？

A: **不适合场景**：
1. **极端长上下文（> 64K）**：推理侧外推难以维持质量，必须继续预训练
2. **长程依赖关键任务**：如多文档推理、长代码理解，外推会破坏依赖建模
3. **精度敏感场景**：医疗/法律等领域，外推带来的精度损失不可接受
4. **频繁长文本**：外推会降低吞吐，不如训练长窗口模型

**适合场景**：
- 偶发长文本需求
- 对精度要求不高（如闲聊、摘要）
- 快速验证原型

### 6. Q: 多轮对话的 position id 如何处理？

A: **三种策略**：
1. **累积 position**（最常用）：
   ```python
   # 第一轮：[0, 1, 2, ..., n]
   # 第二轮：[n+1, n+2, ..., n+m]
   # 问题：超过窗口后无法继续
   ```
   
2. **窗口滑动 + 重映射**：
   ```python
   # 保留最近 W 个 token
   # position id 重置为 [0, 1, ..., W-1]
   # 问题：历史信息丢失
   ```
   
3. **分段 position + cache**（vLLM 支持）：
   ```python
   # 使用 prefix cache 保留历史
   # 每轮从 position=0 开始，通过 cache 复用
   # 优点：无限对话长度；缺点：依赖 cache 命中
   ```

**推荐**：业务场景用累积 position + 滑动窗口兜底；研究场景用 prefix cache。

### 7. Q: 如何评估长上下文模型的性能？

A: **评估体系**：
1. **"大海捞针"测试**：
   - 在长度 L 的文本中插入关键信息
   - 测试不同位置（前/中/后）的检索准确率
   - 指标：准确率 vs 文本长度

2. **LongBench benchmark**：
   - 多任务评估（QA/摘要/代码/Few-shot）
   - 长度覆盖 1K-32K

3. **困惑度曲线**：
   - 绘制 PPL vs 长度曲线
   - 外推阈值：PPL 突然上升的点

4. **短文本回归测试**：
   - MMLU/C-Eval 等标准 benchmark
   - 确保短文本能力未退化

**关键指标**：
- 检索准确率（"大海捞针"）：> 95%
- 短文本性能下降：< 5%
- 困惑度增长：< 20%（在目标长度）

### 8. Q: 长上下文训练遇到 loss 震荡怎么办？

A: **排查步骤**：
1. **数据问题**：
   - 检查长文本数据质量（异常长/重复）
   - 检查 packing 是否导致样本长度极端不均
   - 解决：数据清洗 + 长度分桶

2. **梯度问题**：
   - 长序列梯度累积大，易爆炸
   - 解决：降低学习率（×0.5）+ 梯度裁剪（norm=1.0）

3. **混合精度问题**：
   - 长序列 attention 计算易溢出
   - 解决：使用 BF16（非 FP16）+ loss scaling

4. **批次大小问题**：
   - 长 batch 导致不稳定
   - 解决：减小 batch size，增加 gradient accumulation

**典型配置**：
```python
training_args = {
    "learning_rate": 5e-6,  # 比短文本低
    "gradient_accumulation_steps": 8,  # 模拟大 batch
    "max_grad_norm": 1.0,
    "fp16": False,
    "bf16": True,  # 关键：BF16 更稳定
}
```

## 常见错误（5 个）

### 1. 错误：直接推理长文本不调整位置编码

**现象**：将 4K 训练的模型直接用于 32K 输入，PPL 飙升，检索失败。

**正确做法**：
```python
# 错误：直接加载
model = AutoModelForCausalLM.from_pretrained("model-4k")

# 正确：配置 RoPE scaling
config = AutoConfig.from_pretrained("model-4k")
config.rope_scaling = {"type": "yarn", "factor": 8.0}
config.max_position_embeddings = 32768
model = AutoModelForCausalLM.from_pretrained("model-4k", config=config)
```

### 2. 错误：忽略显存峰值导致 OOM

**现象**：以为 KV cache 能放下，结果 prefill 阶段显存峰值远超 KV cache 大小。

**原因**：FlashAttention 虽避免存储完整注意力矩阵，但 prefill 阶段仍有中间缓冲区，长序列峰值可能达到 KV cache 的 2-3 倍。

**正确做法**：
```python
# 预估显存（包含峰值）
def estimate_peak_memory(model_size_gb, seq_len, batch_size=1):
    # KV cache
    kv_cache = model_size_gb * 2 * seq_len / 4096  # 相对于 4K 的倍数
    
    # Prefill 峰值（经验值）
    peak_factor = 2.5
    peak_memory = model_size_gb + kv_cache * peak_factor
    
    return peak_memory

# 7B 模型，32K 序列
estimate_peak_memory(14, 32768)  # ≈ 50GB
```

### 3. 错误：外推后不验证短文本性能

**现象**：YaRN 外推后长文本效果好，但线上短文本任务投诉激增。

**原因**：外推方法会改变注意力分布，影响短文本建模能力。

**正确做法**：
```python
# 上线前必须测试
def validate_model(model, tokenizer):
    # 1. 短文本 benchmark
    short_text_perf = evaluate_on_benchmark(model, tokenizer, "MMLU")
    
    # 2. 长文本检索
    long_text_perf = needle_in_haystack_test(model, tokenizer, length=16000)
    
    # 3. 回归测试
    assert short_text_perf > baseline * 0.95  # 下降不超过 5%
    assert long_text_perf > 0.9  # 检索准确率 > 90%
    
    return {"short": short_text_perf, "long": long_text_perf}
```

### 4. 错误：混淆 RoPE scaling 与 max_position_embeddings

**现象**：修改了 `rope_scaling` 但忘记更新 `max_position_embeddings`，导致模型仍截断到原长度。

**正确做法**：
```python
config.rope_scaling = {"type": "yarn", "factor": 8.0}
config.max_position_embeddings = 32768  # 必须同步修改！
```

**原理**：`rope_scaling` 只调整位置编码计算，`max_position_embeddings` 控制模型实际接受的输入长度。

### 5. 错误：KV cache 量化后不测试长文本精度

**现象**：INT8 KV cache 在短文本测试正常，长文本检索准确率断崖下跌。

**原因**：长序列下量化误差累积，KV cache 精度影响放大。

**正确做法**：
```python
# 量化前后对比
def test_kv_cache_quantization(model, tokenizer):
    # FP16 KV cache
    model.config.kv_cache_dtype = "fp16"
    fp16_perf = long_context_test(model, tokenizer, length=32000)
    
    # INT8 KV cache
    model.config.kv_cache_dtype = "int8"
    int8_perf = long_context_test(model, tokenizer, length=32000)
    
    # 精度损失检查
    degradation = (fp16_perf - int8_perf) / fp16_perf
    assert degradation < 0.05  # 下降不超过 5%
    
    return {"fp16": fp16_perf, "int8": int8_perf}
```

**题目 1：实现线性插值 RoPE scaling**

```python
import torch

def apply_linear_rope_scaling(position_ids, original_max_len, target_max_len):
    """
    实现线性插值：缩放位置索引
    
    Args:
        position_ids: [batch, seq_len], 原始位置索引
        original_max_len: 训练时的最大长度
        target_max_len: 目标扩展长度
    
    Returns:
        scaled_position_ids: [batch, seq_len], 缩放后的位置索引
    """
    # 计算缩放因子
    scaling_factor = original_max_len / target_max_len
    
    # 线性插值
    scaled_position_ids = (position_ids.float() * scaling_factor).long()
    
    return scaled_position_ids

# 测试
position_ids = torch.arange(0, 16000).unsqueeze(0)  # 16K 序列
scaled = apply_linear_rope_scaling(position_ids, 4096, 16384)
print(f"原始位置范围: [{position_ids.min()}, {position_ids.max()}]")
print(f"缩放后范围: [{scaled.min()}, {scaled.max()}]")
```

**题目 2：计算 KV cache 显存占用**

```python
def estimate_kv_cache_memory(
    num_layers: int,
    num_heads: int,
    head_dim: int,
    seq_len: int,
    dtype_bytes: int = 2,  # FP16 = 2 bytes
    batch_size: int = 1,
):
    """
    估算 KV cache 显存占用
    
    Args:
        num_layers: 层数
        num_heads: 注意力头数
        head_dim: 每个头的维度
        seq_len: 序列长度
        dtype_bytes: 数据类型字节数（FP16=2, BF16=2, INT8=1）
        batch_size: 批次大小
    
    Returns:
        显存占用（GB）
    """
    # 每层 KV cache: 2 (K+V) × num_heads × head_dim × seq_len
    cache_per_layer = 2 * num_heads * head_dim * seq_len
    
    # 总 cache: layers × batch
    total_cache = num_layers * cache_per_layer * batch_size
    
    # 字节数转 GB
    memory_gb = total_cache * dtype_bytes / (1024 ** 3)
    
    return memory_gb

# 示例：Qwen-7B, 32K 序列
memory = estimate_kv_cache_memory(
    num_layers=32,
    num_heads=32,
    head_dim=128,
    seq_len=32768,
    dtype_bytes=2,
    batch_size=1,
)
print(f"KV cache 显存: {memory:.2f} GB")  # 约 16GB
```

**题目 3：实现"大海捞针"测试**

```python
def needle_in_haystack_test(
    model,
    tokenizer,
    context_length: int = 16000,
    needle: str = "重要信息：今晚8点开会",
    needle_position: str = "middle",  # "start", "middle", "end"
):
    """
    大海捞针测试：在长文本中插入关键信息，测试检索能力
    
    Returns:
        bool: 是否成功检索到关键信息
    """
    # 构造填充文本
    filler_text = "这是一段普通的文本内容，用于填充上下文窗口。"
    tokens_per_filler = len(tokenizer.encode(filler_text))
    
    # 计算需要的填充次数
    num_fillers = context_length // tokens_per_filler
    filler = filler_text * num_fillers
    
    # 插入 needle
    filler_tokens = tokenizer.encode(filler)
    if needle_position == "start":
        insert_pos = len(filler_tokens) // 10
    elif needle_position == "middle":
        insert_pos = len(filler_tokens) // 2
    else:  # end
        insert_pos = len(filler_tokens) * 9 // 10
    
    text_tokens = filler_tokens[:insert_pos] + tokenizer.encode(needle) + filler_tokens[insert_pos:]
    text_tokens = text_tokens[:context_length]  # 截断到目标长度
    
    # 构造问题
    question = "这段文本中提到的重要信息是什么？请准确复述。"
    full_text = tokenizer.decode(text_tokens) + "\n\n" + question
    
    # 生成答案
    inputs = tokenizer(full_text, return_tensors="pt").to(model.device)
    outputs = model.generate(**inputs, max_new_tokens=100, temperature=0.1)
    response = tokenizer.decode(outputs[0][inputs["input_ids"].shape[1]:], skip_special_tokens=True)
    
    # 检查是否包含关键信息
    success = "今晚8点开会" in response
    
    return success, response
```

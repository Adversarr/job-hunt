# 训练精度选择（FP16/BF16/FP32）

## 一句话结论
BF16 因动态范围大、训练稳定，已成为大模型混合精度训练的首选；FP16 精度更高但易溢出需 loss scaling；关键算子（softmax/layernorm/累加）保持 FP32 避免数值问题。

## 核心定义/公式

### 数值表示范围
| 精度 | 符号位 | 指数位 | 尾数位 | 动态范围 | 精度 |
|------|--------|--------|--------|----------|------|
| FP32 | 1 | 8 | 23 | ±3.4e38 | 高（约7位有效数字）|
| FP16 | 1 | 5 | 10 | ±65504 | 中（约3位有效数字）|
| BF16 | 1 | 8 | 7 | ±3.4e38 | 低（约2位有效数字）|

### 关键公式

**1. FP16 溢出阈值**
```
FP16 最大值 = 65504
梯度 > 65504 → overflow → inf
梯度 < 6e-8 → underflow → 0
```

**2. Loss Scaling（FP16 训练必需）**
```python
# PyTorch AMP 实现
scaler = torch.cuda.amp.GradScaler(
    init_scale=2**16,      # 初始缩放因子（65536）
    growth_factor=2.0,     # 无溢出时放大倍数
    backoff_factor=0.5,    # 溢出时缩小倍数
    growth_interval=2000   # 无溢出持续步数后放大
)

# 前向传播
with torch.cuda.amp.autocast(dtype=torch.float16):
    loss = model(input)

# 反向传播（自动 scale）
scaler.scale(loss).backward()
scaler.step(optimizer)
scaler.update()
```

**3. BF16 无需 Loss Scaling**
```python
# BF16 混合精度（PyTorch >= 1.10）
with torch.cuda.amp.autocast(dtype=torch.bfloat16):
    loss = model(input)
    # 无需 scaler，BF16 动态范围与 FP32 相同
loss.backward()
optimizer.step()
```

**4. 关键算子保持 FP32**
```python
# PyTorch AMP 自动处理
# 以下操作默认保持 FP32：
# - Softmax（数值稳定性）
# - LayerNorm / RMSNorm（归一化）
# - Loss 计算（交叉熵累加）
# - 梯度累加

# 手动控制示例
with torch.cuda.amp.autocast(enabled=False):
    # 强制 FP32 计算
    norm = torch.rsqrt(x.pow(2).mean(-1, keepdim=True) + 1e-6)
```

## 为什么（2-3 个因果链）

### 1. FP16 易溢出但精度高
- **现象**：FP16 训练易出现梯度爆炸/消失、loss NaN
- **根因**：FP16 指数位仅 5 bit，动态范围 [-65504, 65504]，梯度稍大即 overflow
- **结果**：需要 loss scaling 放大梯度避免 underflow，但 scaling 过大又会 overflow，需动态调整

**因果链**：
```
FP16 动态范围小（5 bit 指数位）
→ 梯度 > 65504 时溢出为 inf
→ 梯度 < 6e-8 时下溢为 0
→ 必须引入 loss scaling 机制
→ 增加训练复杂度和超参调试成本
```

### 2. BF16 训练稳定适合大模型
- **现象**：GPT-3、LLaMA、Qwen 等大模型均采用 BF16 训练
- **根因**：BF16 指数位 8 bit（与 FP32 相同），动态范围 ±3.4e38，不会出现梯度溢出
- **结果**：无需 loss scaling，训练流程简单，稳定性显著提升

**因果链**：
```
BF16 动态范围与 FP32 相同（8 bit 指数位）
→ 梯度不会溢出/下溢
→ 无需 loss scaling
→ 训练稳定性提升 + 超参调试简化
→ 成为当前大模型训练标准选择
```

### 3. 混合精度必须保留关键算子的 FP32
- **现象**：直接全 FP16/BF16 会导致精度损失甚至训练崩溃
- **根因**：softmax 指数运算、layernorm 归一化、loss 累加等操作对数值精度极度敏感
- **结果**：框架自动在关键路径保持 FP32，用户无需手动干预

**因果链**：
```
Softmax 需要计算 exp(x_i) / Σexp(x_j)
→ FP16 精度低，exp 运算误差放大
→ 分母累加精度不足导致数值不稳定
→ 必须在 softmax/layernorm/累加保持 FP32
→ PyTorch AMP 自动处理，用户透明
```

### 代码示例：完整训练 Loop
```python
import torch
import torch.nn as nn
from torch.cuda.amp import autocast, GradScaler

def train_with_amp(model, dataloader, optimizer, epochs, use_bf16=True):
    """
    混合精度训练完整示例
    """
    device = torch.device("cuda")
    model = model.to(device)
    
    # 选择精度策略
    dtype = torch.bfloat16 if use_bf16 else torch.float16
    scaler = None if use_bf16 else GradScaler(init_scale=2**16)
    
    # 检查硬件支持
    if use_bf16 and not torch.cuda.is_bf16_supported():
        raise RuntimeError(
            "BF16 not supported on this GPU. "
            "Use FP16 or upgrade to Ampere+ (A100/RTX3090+)"
        )
    
    model.train()
    for epoch in range(epochs):
        for step, batch in enumerate(dataloader):
            # 数据移到 GPU
            input_ids = batch["input_ids"].to(device)
            labels = batch["labels"].to(device)
            
            optimizer.zero_grad()
            
            # 前向传播（自动混合精度）
            with autocast(dtype=dtype):
                outputs = model(input_ids)
                
                # Loss 计算保持 FP32（autocast 自动处理）
                loss = nn.functional.cross_entropy(
                    outputs.view(-1, outputs.size(-1)),
                    labels.view(-1),
                    ignore_index=-100
                )
            
            # 反向传播
            if scaler is not None:
                # FP16: 需要 scaler
                scaler.scale(loss).backward()
                scaler.unscale_(optimizer)  # 解除 scaling 以便梯度裁剪
                torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
                scaler.step(optimizer)
                scaler.update()
            else:
                # BF16: 无需 scaler
                loss.backward()
                torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
                optimizer.step()
            
            # 监控
            if step % 100 == 0:
                print(f"Epoch {epoch}, Step {step}, Loss {loss.item():.4f}")
                
                # 检查 NaN/Inf
                if torch.isnan(loss) or torch.isinf(loss):
                    print("WARNING: Loss is NaN/Inf, check data or precision!")
                    break

# 使用示例
model = MyTransformerModel()
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4, weight_decay=0.01)
train_with_amp(model, train_loader, optimizer, epochs=3, use_bf16=True)
```

## 权衡分析

| 方案 | 收益 | 代价 | 适用边界 |
|------|------|------|----------|
| **BF16 混合精度** | 训练稳定、无需 loss scaling、显存省 50% | 精度稍低（尾数 7 bit）、需 Ampere+ GPU | 大模型预训练/SFT、A100/H100/RTX3090+ |
| **FP16 混合精度** | 兼容性好（V100/T4 支持）、显存省 50%、精度略高 | 需要 loss scaling、超参调试成本、易溢出 | 兼容性优先场景、旧 GPU、调试阶段 |
| **FP32 全精度** | 精度最高、无数值问题、调试友好 | 显存占用 2x、训练慢、通信开销大 | 小模型（<1B）、调试、数值敏感任务 |
| **INT8 训练** | 显存最省（仅 FP32 的 25%）、吞吐最高 | 精度损失明显、需要量化感知训练、实现复杂 | 边缘部署训练、资源极度受限场景 |

**决策树**：
```
是否有 Ampere+ GPU (A100/RTX3090)?
├─ 是 → 使用 BF16 混合精度
└─ 否 → 检查是否支持 FP16
    ├─ 支持 → 使用 FP16 + GradScaler
    └─ 不支持 → 使用 FP32（CPU 训练或换 GPU）

模型规模?
├─ >10B → 必须混合精度（BF16 优先）
├─ 1-10B → 推荐混合精度
└─ <1B → 可选 FP32（精度优先）或混合精度（速度优先）
```

## 高频追问（至少 5 个）

### 1. Q: BF16 和 FP16 的核心差异是什么？为什么大模型更倾向用 BF16？
**A**: 核心差异在**指数位宽度**：BF16 有 8 bit 指数位（动态范围 ±3.4e38），FP16 仅 5 bit（动态范围 ±65504）。大模型训练梯度范围大，FP16 易溢出需 loss scaling，BF16 动态范围与 FP32 相同，训练更稳定。精度上 FP16 尾数位 10 bit 优于 BF16 的 7 bit，但对训练影响小于动态范围问题。

### 2. Q: 混合精度训练中，哪些算子必须保持 FP32？为什么？
**A**: 三类算子必须保持 FP32：
1. **Softmax**：exp 运算精度敏感，FP16/BF16 累加误差导致数值不稳定
2. **LayerNorm/RMSNorm**：归一化涉及方差计算，精度不足导致梯度爆炸
3. **Loss 累加**：Cross-Entropy 的 sum over tokens，FP16 累加精度损失明显

PyTorch AMP 自动处理这些算子，无需手动干预。

### 3. Q: FP16 训练为什么需要 Loss Scaling？原理是什么？
**A**: FP16 动态范围小，小梯度（< 6e-8）会下溢为 0，导致模型无法学习。Loss Scaling 通过在前向传播后放大 loss，使得梯度相应放大，避免下溢；反向传播后再缩小参数更新。动态调整机制（如 GradScaler）根据梯度溢出情况自动调整 scale factor。

### 4. Q: 如何判断 GPU 是否支持 BF16？不支持 BF16 怎么办？
**A**: 使用 `torch.cuda.is_bf16_supported()` 检查，底层要求 GPU Compute Capability ≥ 8.0（Ampere 架构，如 A100/RTX3090）。不支持时：
- 降级到 FP16 + GradScaler（V100/T4 可用）
- 或使用 FP32（显存充足时）
- 或升级 GPU 硬件

### 5. Q: 混合精度训练相比 FP32 能省多少显存？计算公式是什么？
**A**: 显存节省约 **50%**（仅考虑参数和梯度）。
计算公式：
```
FP32 显存 = 参数(4B) + 梯度(4B) + 优化器状态(8B) = 16B per param
混合精度显存 = 参数(2B) + 梯度(2B) + 优化器状态(8B) = 12B per param
节省比例 = (16 - 12) / 16 = 25%（未计 activation）

实际节省取决于：
- Activation checkpointing 使用情况
- Batch size 和 sequence length
- 通常总体节省 40-50%
```

### 6. Q: 训练时出现 NaN/Inf，如何排查是精度问题？
**A**: 排查顺序：
1. **检查数据**：是否有异常值（inf/nan 输入）
2. **检查学习率**：过大导致梯度爆炸
3. **检查精度策略**：
   - FP16: Loss scale 是否过大/过小？尝试调整 `init_scale`
   - BF16: 检查是否误用 FP16（硬件不支持自动降级）
4. **检查关键算子**：softmax/layernorm 是否被强制低精度
5. **调试手段**：
   ```python
   # 监控梯度
   for name, param in model.named_parameters():
       if param.grad is not None:
           if torch.isnan(param.grad).any():
               print(f"NaN gradient in {name}")
   ```

### 7. Q: KV Cache 应该用什么精度？FP16/BF16/INT8？
**A**: 推理时 KV Cache 精度选择：
- **FP16/BF16**：首选，精度损失小，显存节省 50%
- **INT8**：显存最省（75%），但精度损失明显，长文本场景更敏感
- **实践**：先用 FP16/BF16，显存不足再考虑 INT8 量化（需评估精度损失）

训练时 KV Cache 通常不存储（teacher forcing），推理时精度选择见 [[02-推理量化路线]]。

## 常见错误（至少 3 个）

### 1. 错误：在不支持 BF16 的 GPU 上强制使用 BF16
**现象**：程序崩溃或训练极慢
**原因**：V100/T4 等 GPU 不支持 BF16，`autocast(dtype=torch.bfloat16)` 会降级到 FP32，失去混合精度收益
**正确做法**：
```python
# 错误
with autocast(dtype=torch.bfloat16):  # V100 上会失败
    loss = model(x)

# 正确
dtype = torch.bfloat16 if torch.cuda.is_bf16_supported() else torch.float16
with autocast(dtype=dtype):
    loss = model(x)
```

### 2. 错误：FP16 训练忘记使用 GradScaler
**现象**：Loss 不下降或出现 NaN
**原因**：FP16 梯度下溢为 0，参数无法更新
**正确做法**：
```python
# 错误
with autocast(dtype=torch.float16):
    loss = model(x)
loss.backward()  # 梯度下溢
optimizer.step()

# 正确
scaler = GradScaler()
with autocast(dtype=torch.float16):
    loss = model(x)
scaler.scale(loss).backward()
scaler.step(optimizer)
scaler.update()
```

### 3. 错误：手动将所有算子转为 FP16，包括 softmax/layernorm
**现象**：训练不稳定、loss NaN、精度下降严重
**原因**：关键算子对精度敏感，强制低精度导致数值问题
**正确做法**：
```python
# 错误
model.half()  # 强制所有参数 FP16，包括 layernorm

# 正确
# 使用 autocast 自动处理，框架会保留关键算子的 FP32
with autocast(dtype=torch.bfloat16):
    loss = model(x)
```

### 4. 错误：BF16 和 FP16 混用未检查硬件支持
**现象**：在不同 GPU 上表现不一致，V100 上崩溃
**原因**：未做硬件兼容性检查，代码可移植性差
**正确做法**：
```python
# 正确：统一检查入口
def get_autocast_dtype():
    if torch.cuda.is_bf16_supported():
        return torch.bfloat16
    else:
        return torch.float16

dtype = get_autocast_dtype()
with autocast(dtype=dtype):
    loss = model(x)
```

### 5. 错误：梯度裁剪在 scaler 之前执行
**现象**：梯度裁剪失效，梯度爆炸
**原因**：GradScaler 会放大梯度，在 scale 之前裁剪会低估实际梯度范数
**正确做法**：
```python
# 错误
scaler.scale(loss).backward()
torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)  # 裁剪的是 scaled 梯度
scaler.step(optimizer)

# 正确
scaler.scale(loss).backward()
scaler.unscale_(optimizer)  # 先解除 scaling
torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)  # 裁剪真实梯度
scaler.step(optimizer)
```

```python
    with torch.cuda.amp.autocast(dtype=dtype):
        loss = model(batch)
    
    if scaler is not None:  # FP16
        scaler.scale(loss).backward()
        scaler.unscale_(optimizer)
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        scaler.step(optimizer)
        scaler.update()
    else:  # BF16
        loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        optimizer.step()
    
    return loss.item()
```

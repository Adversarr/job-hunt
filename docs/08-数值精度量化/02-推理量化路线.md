# 推理量化路线（INT8/INT4）

## 一句话结论
INT8/INT4 量化通过降低数值精度节省显存（INT8 省 2x，INT4 省 4x）并加速推理，但会引入精度损失；INT8 适合大多数层，INT4 对 outlier 敏感需分组量化+混合精度保关键层，KV cache 可量化到 INT8 但长文本下精度问题更明显。

## 核心定义/公式

### 量化基本公式
```python
# 对称量化（symmetric quantization）
Q(x) = round(x / scale)
scale = max(|x|) / (2^(b-1) - 1)  # b 为量化位数

# 非对称量化（asymmetric quantization）
Q(x) = round(x / scale) + zero_point
scale = (max(x) - min(x)) / (2^b - 1)
zero_point = round(-min(x) / scale)

# 反量化（dequantization）
x_hat = (Q(x) - zero_point) * scale
```

### INT8 vs INT4 量化范围
```python
# INT8 对称量化
int8_min, int8_max = -128, 127
# FP16 值域：[-65504, 65504]
# BF16 值域：[-3.4e38, 3.4e38]（但精度只有 8 位尾数）

# INT4 对称量化
int4_min, int4_max = -8, 7
# 可表示范围更小，对 outlier 敏感度更高
```

### 权重量化 vs 激活量化
```python
# 权重量化（Weight Quantization）
# - 离线量化，静态 scale（权重固定）
# - per-channel 或 per-group 量化
# - 误差可控，易优化

# 激活量化（Activation Quantization）
# - 在线量化，动态 scale（激活值变化）
# - per-tensor 或 per-token 量化
# - 误差难控制，需校准数据

# W8A8：权重和激活都量化到 INT8
# W4A16：权重 INT4，激活 FP16（常见于 INT4 推理）
```

### 分组量化（Group-wise Quantization）
```python
# GPTQ/AutoGPTQ 参数
group_size = 128  # 每 128 个权重共享一个 scale
# 权重形状：[out_features, in_features]
# 每组：[out_features, group_size] 共享 scale/zero_point

# 计算示例
weight_group = weight[:, i*group_size:(i+1)*group_size]
scale_i = weight_group.abs().max() / 7  # INT4 对称量化
quant_group = (weight_group / scale_i).round().clamp(-8, 7)
```

## 为什么（2-3 个因果链）

### 1. 为什么需要量化：显存/带宽瓶颈 → 量化降精度
```
现象：大模型推理显存占用大、吞吐低
  ↓
根因1：显存瓶颈
  - 70B 模型 FP16 权重：140GB
  - 单卡 A100 80GB 放不下
  - 需要模型并行或多卡推理
  ↓
根因2：内存带宽瓶颈（decode 阶段）
  - decode 每步生成 1 token
  - 需要读取全部权重到 cache
  - 带宽成为瓶颈（A100: 2TB/s，H100: 3.35TB/s）
  ↓
结果：量化降低存储需求
  - INT8：显存减半，带宽需求减半
  - INT4：显存降至 1/4，带宽需求降至 1/4
  - 推理速度提升 2-4x（带宽受限场景）
```

### 2. 为什么 INT4 损失更大：精度范围小 → outlier 敏感 → 误差放大
```
现象：INT4 量化后质量明显下降
  ↓
根因1：量化粒度更粗
  - INT8：255 个离散值（对称模式）
  - INT4：15 个离散值（对称模式）
  - 量化误差：INT4 ≈ 16x INT8
  ↓
根因2：outlier 权重/激活更敏感
  - 大模型存在 activation outlier（部分维度激活值特别大）
  - 这些 outlier 对 scale 影响大
  - INT4 量化后，非 outlier 区域精度损失严重
  ↓
根因3：关键层误差累积
  - 输出层（lm_head）：直接影响 logit 分布
  - LayerNorm：影响后续层输入分布
  - 注意力投影：影响注意力分数
  ↓
结果：INT4 质量下降明显
  - MMLU：INT8 损失 1-2%，INT4 损失 3-5%
  - 长文本任务：INT4 损失更严重
  - 需要分组量化 + 混合精度补救
```

### 3. 为什么有的层不能乱量化：敏感路径 → 误差累积 → 质量骤降
```
现象：某些层量化后质量崩溃
  ↓
根因1：LayerNorm/RMSNorm 的统计敏感性
  - LayerNorm 计算 mean/variance
  - 量化后统计量偏差 → 后续层输入分布偏移
  - 误差累积到后续所有层
  ↓
根因2：输出层（lm_head）直接影响 token 概率
  - lm_head 输出 logit
  - softmax 前 logit 的微小差异 → 概率分布大变
  - 尤其在 beam search / sampling 时
  ↓
根因3：注意力机制的数值敏感性
  - QK^T 点积 → 缩放 → softmax
  - Softmax 对输入敏感（exp 放大）
  - 量化误差经过 softmax 后可能放大
  ↓
结果：关键层必须保持高精度
  - LayerNorm/RMSNorm：保持 FP16/BF16
  - lm_head：保持 FP16 或单独校准
  - 第一层/最后几层：混合精度
```

### 3. KV Cache 量化

#### KV Cache 显存估算
```python
# KV Cache 显存公式
def estimate_kv_cache_memory(
    batch_size: int,
    seq_len: int,
    num_layers: int,
    num_heads: int,
    head_dim: int,
    dtype_bytes: int = 2  # FP16/BF16 = 2 bytes
) -> int:
    """
    KV Cache 显存估算
    
    Args:
        batch_size: 批大小
        seq_len: 序列长度
        num_layers: 层数
        num_heads: 注意力头数
        head_dim: 每个头的维度
        dtype_bytes: 数据类型字节数
    
    Returns:
        显存占用（GB）
    """
    # K 和 V 两份，每层都需要
    kv_cache_bytes = (
        batch_size * seq_len * 
        num_layers * num_heads * head_dim * 
        2 *  # K 和 V
        dtype_bytes
    )
    return kv_cache_bytes / (1024**3)  # 转换为 GB

# 示例：Qwen2-7B
# num_layers=32, num_heads=32, head_dim=128
# batch_size=32, seq_len=4096, FP16
kv_mem = estimate_kv_cache_memory(32, 4096, 32, 32, 128, 2)
# 结果：约 16GB
```

## 高频追问（至少 5 个）

### 1. Q: INT8 和 INT4 量化的主要区别是什么？
A: 
- **精度损失**：INT8 损失 1-2%，INT4 损失 3-5%
- **对 outlier 敏感度**：INT4 更敏感，需要分组量化
- **显存节省**：INT8 省 2x，INT4 省 4x
- **部署复杂度**：INT8 简单，INT4 需要混合精度或 QLoRA 补救
- **适用场景**：INT8 通用，INT4 适合边缘设备或极致成本优化

### 2. Q: 为什么长上下文下量化精度问题更明显？
A:
```
长上下文 → KV Cache 累积误差 → 注意力分数偏移 → 上下文检索能力下降

具体原因：
1. KV Cache 累积：
   - 序列越长，KV Cache 量化误差累积越多
   - 长序列中，早期 token 的 K/V 误差传播到后续所有位置

2. 注意力机制敏感：
   - 长序列 attention score 更分散
   - 量化误差导致 softmax 前分数偏移
   - top-k attention 受影响

3. 位置编码误差：
   - RoPE 等位置编码依赖精确数值
   - 量化后相对位置信息受损

4. 数值溢出风险：
   - 长序列 attention score 累积更大
   - INT8/INT4 容易溢出或精度不足

实测数据：
- MMLU（短文本）：INT8 损失 1-2%，INT4 损失 3-5%
- LongBench（长文本）：INT8 损失 3-5%，INT4 损失 8-15%
```

### 3. Q: 哪些层不能量化？为什么？
A:
```
必须保持高精度的层：

1. LayerNorm / RMSNorm
   原因：计算 mean/variance，量化后统计量偏差
   影响：误差累积到后续所有层
   建议：保持 FP16/BF16

2. 输出层（lm_head）
   原因：直接影响 logit 分布 → token 概率
   影响：sampling/beam search 质量下降
   建议：保持 FP16 或单独校准

3. 第一层 embedding
   原因：输入层误差传播到所有层
   影响：基础表示质量下降
   建议：保持 FP16

4. 注意力投影中的敏感维度
   原因：activation outlier 集中在部分维度
   影响：关键 token 表示受损
   建议：使用 AWQ 自动识别保护

5. 模型的最后几层
   原因：接近输出，误差无机会修正
   影响：直接反映在生成质量
   建议：混合精度，最后 2-3 层保持 FP16
```

### 4. Q: KV Cache 用 FP16 还是 BF16 有差吗？
A:
```
FP16 vs BF16 用于 KV Cache：

共同点：
- 显存占用相同（都是 2 bytes）
- 都能保存 KV Cache 的主要信息

区别：

1. 动态范围：
   - FP16: [-65504, 65504]
   - BF16: [-3.4e38, 3.4e38]（更大）
   
2. 精度：
   - FP16: 10 位尾数，精度更高
   - BF16: 7 位尾数，精度稍低

3. 稳定性：
   - FP16: 易溢出（activation outlier）
   - BF16: 不易溢出，更稳定

实际影响：
- 短文本（<2048 tokens）：差异小（<0.5%）
- 长文本（>4096 tokens）：
  - FP16: 可能溢出，需要 loss scaling
  - BF16: 更稳定，推荐用于 KV Cache

推荐：
- 训练时用 BF16（稳定）
- KV Cache 推荐 BF16（防溢出）
- 如果模型训练用 FP16，KV Cache 也用 FP16
```

### 5. Q: 如何选择量化方法（GPTQ / AWQ / GGUF）？
A:
```
量化方法对比：

1. GPTQ (AutoGPTQ)
   原理：基于 Hessian 矩阵的逐层量化
   优点：
   - 量化速度快（5-10 分钟）
   - 社区支持好（HuggingFace 集成）
   - 支持 INT8/INT4
   缺点：
   - 精度稍低于 AWQ
   - 需要校准数据
   适用：通用场景，快速量化

2. AWQ (Activation-aware Weight Quantization)
   原理：基于激活值重要性识别，保护关键权重
   优点：
   - 精度更高（比 GPTQ 高 1-2%）
   - 自动识别敏感权重
   - 支持混合精度
   缺点：
   - 量化速度慢（15-30 分钟）
   - 需要校准数据
   适用：高精度要求，质量敏感场景

3. GGUF (llama.cpp)
   原理：CPU/GPU 混合推理优化
   优点：
   - CPU 推理友好
   - 无需 GPU 也能部署
   - 支持多种量化级别（Q4_0/Q4_K_M/Q5_K_M 等）
   缺点：
   - GPU 推理速度不如 vLLM
   - 部分算子未优化
   适用：边缘设备，CPU 推理，资源受限场景

选择建议：
- GPU 推理 + 速度优先：GPTQ + vLLM
- GPU 推理 + 精度优先：AWQ + vLLM
- CPU 推理 / 边缘设备：GGUF + llama.cpp
- 快速验证：GPTQ（量化快）
```

### 6. Q: QLoRA 如何补救 INT4 量化的精度损失？
A:
```
QLoRA 补救机制：

1. 原理：
   - INT4 量化模型作为"冻结基底"
   - 添加 FP16/BF16 的 LoRA 适配器
   - 微调 LoRA 参数适配下游任务

2. 技术细节：
   - NF4 量化（NormalFloat4）：
     * 针对正态分布权重优化
     * 比 INT4 精度更高
   
   - Double Quantization：
     * 量化的 scale 也量化一次
     * 进一步节省显存（约 0.5GB/65B）
   
   - Paged Optimizer：
     * 优化器状态存 CPU
     * GPU 显存进一步降低

3. 精度恢复效果：
   - 无微调：INT4 损失 3-5%
   - QLoRA 微调后：
     * 下游任务质量恢复 80-90%
     * 通用能力恢复 70-80%

4. 关键参数：
   lora_r=64          # LoRA 秩（越大容量越大）
   lora_alpha=16      # 缩放系数（alpha/r）
   target_modules=[   # 适配器位置
       "q_proj", "k_proj", 
       "v_proj", "o_proj",
       "gate_proj", "up_proj", "down_proj"
   ]

5. 成本：
   - 训练数据：1000-10000 条高质量样本
   - 训练时间：1-3 小时（7B 模型）
   - 显存：单卡 24GB 可微调 7B 模型
```

### 7. Q: 量化模型的推理速度一定能提升吗？
A:
```
不一定，取决于瓶颈类型：

1. 带宽受限场景（decode 阶段常见）：
   - 特征：权重读取时间 > 计算时间
   - 量化收益：显著（2-4x）
   - 原因：权重变小，带宽需求降低
   
2. 计算受限场景（prefill 大 batch）：
   - 特征：计算时间 > 权重读取时间
   - 量化收益：有限（1.2-1.5x）
   - 原因：计算量未减少（INT8/INT4 计算仍需反量化）

3. 内存受限场景：
   - 特征：显存不足，需要 offload
   - 量化收益：显著（能装下才能推理）

实测数据（A100，Qwen2-7B）：
- decode 阶段（batch=1）：
  * INT8: 1.8x 加速
  * INT4: 2.5x 加速
  
- prefill 阶段（batch=32）：
  * INT8: 1.3x 加速
  * INT4: 1.5x 加速

- 显存受限场景（70B 模型）：
  * FP16: 单卡无法推理
  * INT4: 单卡可推理，速度从 0 → 可用

结论：
- decode 速度提升明显（带宽受限）
- prefill 速度提升有限（计算受限）
- 显存瓶颈场景是"从无到有"的质变
```

### 8. Q: 如何评估量化对模型质量的影响？
A:
```
量化质量评估框架：

1. 自动评测（Benchmark）：
   - 通用能力：
     * MMLU（知识问答）
     * HellaSwag（常识推理）
     * ARC（科学推理）
     * PIQA（物理常识）
   
   - 长文本能力：
     * LongBench（长文本理解）
     * Needle in Haystack（大海捞针）
     * L-Eval（长文本评估）
   
   - 对比指标：
     * FP16 baseline vs INT8/INT4
     * 相对损失百分比

2. 下游任务评测：
   - 业务相关任务（如代码生成、对话质量）
   - 人工评估（偏好度、可用性）
   - A/B 测试（线上效果）

3. 数值稳定性检测：
   - 检查 NaN/Inf 出现频率
   - 观察 softmax 前的 score 分布
   - 监控 generation diversity

4. 典型损失范围：
   MMLU:
   - FP16: baseline
   - INT8: -1~2%
   - INT4: -3~5%
   - INT4 + QLoRA: -1~2%
   
   LongBench:
   - FP16: baseline
   - INT8: -3~5%
   - INT4: -8~15%
   - INT4 + QLoRA: -3~5%

5. 可接受性判断：
   - 损失 < 2%：可接受
   - 损失 2-5%：需权衡成本
   - 损失 > 5%：不推荐（除非极端成本限制）
```

## 常见错误（至少 3 个）

### 1. 错误：对所有层无差别量化
**描述**：直接对整个模型量化，不区分敏感层和非敏感层。

**后果**：
- LayerNorm 量化后，后续层输入分布偏移
- 输出层量化后，logit 分布异常
- 质量骤降 5-10%

**正确做法**：
```python
# AutoGPTQ 排除敏感层
from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig

quantize_config = BaseQuantizeConfig(
    bits=4,
    group_size=128
)

# 方法1：通过 quantize_config 的 exclude_modules
gptq_model = AutoGPTQForCausalLM.from_quantized(
    model_path,
    quantize_config=quantize_config,
    exclude_modules=["lm_head", "model.norm"]  # 排除敏感层
)

# 方法2：手动设置某些层为 FP16
for name, param in gptq_model.named_parameters():
    if "lm_head" in name or "norm" in name:
        param.data = param.data.half()  # 保持 FP16
```

### 2. 错误：INT4 不使用分组量化
**描述**：INT4 量化时 group_size 设置过大或不分组（per-tensor）。

**后果**：
- outlier 权重拉大 scale
- 大部分权重精度损失严重
- 质量下降 5-8%

**正确做法**：
```python
# 错误配置
quantize_config = BaseQuantizeConfig(
    bits=4,
    group_size=-1  # per-tensor，不分组，精度差
)

# 正确配置
quantize_config = BaseQuantizeConfig(
    bits=4,
    group_size=128  # 推荐 64/128，平衡精度和模型大小
)

# group_size 选择指南：
# - 32: 精度最高，量化模型最大
# - 64: 平衡选择
# - 128: 默认推荐，性价比高
# - 256: 模型最小，精度稍差
```

### 3. 错误：KV Cache INT8 用于短文本场景
**描述**：不加区分地对所有场景使用 KV Cache INT8 量化。

**后果**：
- 短文本场景精度损失 1-2%，但收益有限（显存节省不多）
- 额外的量化/反量化开销
- 得不偿失

**正确做法**：
```python
from vllm import LLM

# 短文本场景（seq_len < 2048）
llm_short = LLM(
    model="Qwen/Qwen2-7B",
    kv_cache_dtype="auto"  # 保持 FP16，不量化
)

# 长文本场景（seq_len > 4096）
llm_long = LLM(
    model="Qwen/Qwen2-7B",
    kv_cache_dtype="int8",  # 量化 KV Cache
    max_model_len=8192      # 支持更长上下文
)

# 判断依据：
# 1. 序列长度 > 2048：考虑 KV Cache INT8
# 2. 序列长度 < 1024：不推荐量化
# 3. 显存是否是瓶颈：是 → 量化；否 → 不量化
```

### 4. 错误：使用不兼容的量化工具链
**描述**：用 GPTQ 量化，但用 TensorRT-LLM 推理（或反之）。

**后果**：
- 模型加载失败
- 需要重新量化，浪费时间
- 不同工具链的量化格式不兼容

**正确做法**：
```python
# 工具链搭配：

# 方案1：AutoGPTQ + vLLM
# 量化
from auto_gptq import AutoGPTQForCausalLM
gptq_model = AutoGPTQForCausalLM.from_quantized(...)
gptq_model.save_quantized("./model-gptq")

# 推理
from vllm import LLM
llm = LLM(model="./model-gptq", quantization="gptq")

# 方案2：AWQ + vLLM
# 量化
from awq import AutoAWQForCausalLM
awq_model = AutoAWQForCausalLM.from_pretrained(...)
awq_model.quantize(...)
awq_model.save_quantized("./model-awq")

# 推理
from vllm import LLM
llm = LLM(model="./model-awq", quantization="awq")

# 方案3：GGUF + llama.cpp
# 量化
# 使用 llama.cpp 的 convert.py 脚本
# python convert.py ./model --outtype q4_k_m --outfile ./model-q4km.gguf

# 推理
# ./main -m ./model-q4km.gguf -p "你好"

# 不要混用：
# ✗ GPTQ 量化 + TensorRT-LLM 推理
# ✗ AWQ 量化 + llama.cpp 推理
# ✓ GPTQ 量化 + vLLM 推理
# ✓ AWQ 量化 + vLLM 推理
# ✓ GGUF 量化 + llama.cpp 推理
```

### 5. 错误：忽略校准数据质量
**描述**：使用随机文本或低质量数据作为量化校准数据。

**后果**：
- 激活量化 scale 不准确
- 质量损失比预期大 2-3%
- 长尾场景表现差

**正确做法**：
```python
# 校准数据要求：
# 1. 数量：128-512 条样本（越多越好，但收益递减）
# 2. 分布：覆盖目标任务分布
# 3. 长度：包含短/中/长文本

# 错误示例
calibration_data = ["测试文本1", "测试文本2"]  # 太少，质量差

# 正确示例
# 从训练集中采样
import random
train_data = load_dataset("your_train_data")
calibration_data = random.sample(train_data, 512)

# 或使用多样化数据
calibration_data = [
    "短文本示例",
    "中等长度的文本示例，包含多个句子...",
    "长文本示例，包含完整的段落或文章...",
    # ... 覆盖不同长度、领域、风格
]

# 量化时传入
gptq_model.quantize(calibration_data)
```

1. **5 分钟能写出的代码/公式**：

   **题目1：实现 INT8 对称量化**
   ```python
   def quantize_int8(weight: torch.Tensor) -> Tuple[torch.Tensor, float]:
       """
       实现对称 INT8 量化
       
       Args:
           weight: [out_features, in_features] 权重矩阵
       
       Returns:
           quant_weight: 量化后的权重（INT8）
           scale: 量化缩放因子
       """
       # 参考答案
       scale = weight.abs().max() / 127.0
       quant_weight = (weight / scale).round().clamp(-128, 127).to(torch.int8)
       return quant_weight, scale
   ```

   **题目2：估算 KV Cache 显存占用**
   ```python
   def estimate_kv_cache(
       batch_size: int,
       seq_len: int,
       num_layers: int,
       num_heads: int,
       head_dim: int,
       dtype: str = "fp16"
   ) -> float:
       """
       估算 KV Cache 显存占用（GB）
       
       Args:
           batch_size: 批大小
           seq_len: 序列长度
           num_layers: 层数
           num_heads: 注意力头数
           head_dim: 每个头的维度
           dtype: 数据类型（"fp16"/"int8"）
       
       Returns:
           显存占用（GB）
       """
       # 参考答案
       dtype_bytes = {"fp16": 2, "bf16": 2, "int8": 1, "int4": 0.5}
       bytes_per_element = dtype_bytes[dtype]
       
       kv_cache_bytes = (
           batch_size * seq_len * 
           num_layers * num_heads * head_dim * 
           2 *  # K 和 V
           bytes_per_element
       )
       return kv_cache_bytes / (1024 ** 3)
   ```

   **题目3：推导量化误差**
   ```
   给定权重 w = [0.1, 0.5, -0.3, 0.8]，实现 INT4 对称量化并计算量化误差。
   
   参考答案：
   1. 计算 scale: max(|w|) = 0.8, scale = 0.8 / 7 = 0.114
   2. 量化: Q(w) = round([0.1, 0.5, -0.3, 0.8] / 0.114) = [1, 4, -3, 7]
   3. 反量化: w_hat = [1, 4, -3, 7] * 0.114 = [0.114, 0.456, -0.342, 0.798]
   4. 量化误差: MSE = mean((w - w_hat)^2) ≈ 0.0006
   ```

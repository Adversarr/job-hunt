# 显存与吞吐估算（微调视角）

## 一句话结论
训练显存 ≈ 参数权重 + 梯度 + 优化器状态 + 激活值；LoRA 相比全参可节省 60-90% 显存，主要省在优化器状态和梯度上，但激活值不变。

## 核心定义/公式

### 显存占用组成（FP16/BF16 混合精度训练）

```
总显存 = 模型参数 + 梯度 + 优化器状态 + 激活值 + 临时缓冲区 + 碎片

具体公式：
- 参数权重：P × 2 bytes (FP16/BF16)
- 梯度：P × 2 bytes (FP16/BF16)
- 优化器状态 (AdamW)：
  - 一阶动量 (m)：P × 4 bytes (FP32)
  - 二阶动量 (v)：P × 4 bytes (FP32)
  - 参数主副本：P × 4 bytes (FP32, 用于更新)
  - 总计：P × 12 bytes
- 激活值：取决于 batch_size × seq_len × hidden_dim × num_layers
```

### 显存估算公式（简化版）

```python
# 全参微调（FP16 混合精度，单卡）
def estimate_full_finetune_memory(
    num_params: int,           # 参数量（单位：B）
    batch_size: int,
    seq_len: int,
    hidden_dim: int,
    num_layers: int,
    activation_checkpointing: bool = False
):
    """
    估算全参微调显存占用
    
    Args:
        num_params: 参数量（单位：Billion）
        batch_size: 批大小
        seq_len: 序列长度
        hidden_dim: 隐藏维度
        num_layers: 层数
        activation_checkpointing: 是否开启激活检查点
    
    Returns:
        dict: 各部分显存占用（单位：GB）
    """
    P = num_params * 1e9  # 转换为参数数量
    
    # 参数权重 (FP16)
    params_mem = P * 2 / 1e9  # GB
    
    # 梯度 (FP16)
    grads_mem = P * 2 / 1e9  # GB
    
    # 优化器状态 (AdamW: m, v, param_master)
    # m: FP32, v: FP32, param_master: FP32
    optimizer_mem = P * 12 / 1e9  # GB
    
    # 激活值（粗略估算）
    # 每层激活值 ≈ batch_size × seq_len × hidden_dim × 每token激活倍数
    # Transformer 每层激活倍数约 10-20（取决于 attention + FFN）
    activation_factor = 15 if not activation_checkpointing else 2
    activation_per_layer = batch_size * seq_len * hidden_dim * activation_factor
    activations_mem = activation_per_layer * num_layers * 2 / 1e9  # FP16
    
    # 临时缓冲区和碎片（约 10-20%）
    temp_and_fragment = 0.15
    
    total = (params_mem + grads_mem + optimizer_mem + activations_mem) * (1 + temp_and_fragment)
    
    return {
        'params': params_mem,
        'grads': grads_mem,
        'optimizer': optimizer_mem,
        'activations': activations_mem,
        'total': total
    }

# LoRA 微调显存估算
def estimate_lora_memory(
    num_params: int,           # 基座模型参数量
    lora_rank: int,            # LoRA rank
    lora_alpha: int,           # LoRA alpha
    target_modules: list,      # 应用 LoRA 的模块
    batch_size: int,
    seq_len: int,
    hidden_dim: int,
    num_layers: int
):
    """
    估算 LoRA 微调显存占用
    
    关键：LoRA 只优化低秩矩阵，冻结基座模型
    """
    P = num_params * 1e9
    
    # 基座模型权重（冻结，FP16）
    base_params_mem = P * 2 / 1e9  # GB
    
    # LoRA 参数量估算（假设对 Q、K、V、O 投影层应用 LoRA）
    # 每层 LoRA 参数 = 2 × hidden_dim × rank (A 矩阵 + B 矩阵)
    # Q、K、V、O 共 4 个投影，每个有 down (A) 和 up (B)
    lora_params_per_layer = 4 * 2 * hidden_dim * lora_rank
    total_lora_params = lora_params_per_layer * num_layers
    lora_params_mem = total_lora_params * 2 / 1e9  # GB
    
    # LoRA 梯度（只对 LoRA 参数）
    lora_grads_mem = total_lora_params * 2 / 1e9  # GB
    
    # LoRA 优化器状态（只对 LoRA 参数）
    lora_optimizer_mem = total_lora_params * 12 / 1e9  # GB
    
    # 激活值（与全参相同，这是关键瓶颈！）
    activation_factor = 15  # 无 checkpointing
    activation_per_layer = batch_size * seq_len * hidden_dim * activation_factor
    activations_mem = activation_per_layer * num_layers * 2 / 1e9
    
    # 缓冲区和碎片
    temp_and_fragment = 0.15
    
    total = (base_params_mem + lora_params_mem + lora_grads_mem + 
             lora_optimizer_mem + activations_mem) * (1 + temp_and_fragment)
    
    return {
        'base_params': base_params_mem,
        'lora_params': lora_params_mem,
        'lora_grads': lora_grads_mem,
        'lora_optimizer': lora_optimizer_mem,
        'activations': activations_mem,
        'total': total
    }
```

### 吞吐估算公式

```python
def estimate_training_throughput(
    num_params: int,           # 参数量（B）
    batch_size: int,
    seq_len: int,
    gpu_flops: int,            # GPU 算力（TFLOPS）
    gpu_memory_bw: int,        # 显存带宽（GB/s）
    mfu: float = 0.4           # Model FLOPS Utilization
):
    """
    估算训练吞吐量（tokens/s）
    
    Args:
        num_params: 模型参数量（Billion）
        batch_size: 批大小
        seq_len: 序列长度
        gpu_flops: GPU 理论峰值算力（例如 A100=312 TFLOPS）
        gpu_memory_bw: 显存带宽（例如 A100=2039 GB/s）
        mfu: 模型算力利用率（实际/理论，典型 0.3-0.6）
    
    Returns:
        tokens_per_second: 每秒处理的 token 数
    """
    P = num_params * 1e9
    
    # 每个 token 的计算量（前向 + 反向）
    # 经验公式：~6 × P FLOPs per token
    flops_per_token = 6 * P
    
    # 实际可用算力
    effective_flops = gpu_flops * 1e12 * mfu
    
    # 理论吞吐（算力受限）
    tokens_per_sec_compute = effective_flops / flops_per_token
    
    # 显存带宽受限（主要用于激活值读写）
    # 粗略估计：每个 token 需读写约 10-20 × hidden_dim bytes
    activation_bytes_per_token = 15 * (P / num_params * 5120)  # 粗略
    tokens_per_sec_memory = gpu_memory_bw * 1e9 / activation_bytes_per_token
    
    # 实际吞吐受限于两者较小值
    tokens_per_second = min(tokens_per_sec_compute, tokens_per_sec_memory)
    
    # 每步时间
    tokens_per_step = batch_size * seq_len
    time_per_step = tokens_per_step / tokens_per_second
    
    return {
        'tokens_per_second': tokens_per_second,
        'time_per_step': time_per_step,
        'samples_per_second': tokens_per_second / seq_len,
        'compute_limited': tokens_per_sec_compute < tokens_per_sec_memory
    }
```

## 为什么（2-3 个因果链）

### 1. 优化器状态是显存大头

**现象**：全参微调时，优化器状态（Adam）占显存 60%+，远超模型参数本身。

**根因**：
- Adam 优化器需要存储两个动量（一阶 m 和二阶 v），每个都是 FP32 精度
- 混合精度训练中，参数有 FP16 和 FP32 两份副本（FP16 用于计算，FP32 用于更新）
- 每个参数需要 2×4+2+4 = 14 bytes（m + v + grad_fp16 + param_fp32 + param_fp16）

**结果**：
- 14B 模型全参微调：参数 28GB → 优化器状态 168GB（6倍）
- ZeRO-3 通过切分优化器状态，可降至 28GB/卡（8卡）
- LoRA 只优化少量参数，优化器状态可忽略

### 2. 激活值随 batch_size 和 seq_len 线性增长

**现象**：增大 batch_size 或 seq_len 时，显存占用急剧上升，且 LoRA 无法缓解。

**根因**：
- 激活值存储反向传播所需的中间结果
- 每层激活值 ∝ batch_size × seq_len × hidden_dim
- Transformer 每层有多个子层（attention、FFN、norm），每层都需存储
- 40 层 × (QK^T、softmax、attn_output、FFN中间态) 累积

**结果**：
- batch_size=4, seq_len=2048 → 激活值 ~50GB
- batch_size=8, seq_len=4096 → 激活值 ~200GB（4倍）
- **Activation Checkpointing** 可牺牲 20-30% 算力，换取 50-70% 激活显存节省
- LoRA 不改变计算图，激活值与全参相同

### 3. INT8 量化在推理省钱，训练受限

**现象**：INT8 量化推理可省 50% 显存，但训练时节省有限。

**根因**：
- **推理时**：只需权重（INT8）+ 激活（INT8/FP16），显存占用大幅下降
- **训练时**：
  - 权重可量化（但需要量化/反量化开销）
  - 梯度需要 FP16（反向传播精度要求）
  - 优化器状态必须 FP32（动量累积精度要求）
  - 激活值通常 FP16（避免精度损失）

**结果**：
- INT8 训练（如 QLoRA）主要节省基座模型权重存储（28GB→14GB）
- 但优化器状态、梯度、激活仍需 FP16/FP32
- QLoRA 真正的价值：单卡加载更大模型（如 70B→35GB 权重），而非训练加速

```python
def forward_with_checkpoint(self, hidden_states):
    def create_custom_forward(module):
        def custom_forward(*inputs):
            return module(*inputs)
```

### 代码示例：实际训练显存监控

```python
import torch
import psutil
import gc

def get_memory_usage():
    """获取当前显存和内存占用"""
    if torch.cuda.is_available():
        gpu_mem = torch.cuda.memory_allocated() / 1e9  # GB
        gpu_mem_cached = torch.cuda.memory_reserved() / 1e9
    else:
        gpu_mem, gpu_mem_cached = 0, 0
    
    cpu_mem = psutil.Process().memory_info().rss / 1e9
    
    return {
        'gpu_allocated': gpu_mem,
        'gpu_reserved': gpu_mem_cached,
        'cpu_memory': cpu_mem
    }

def profile_training_step(model, batch, optimizer, use_lora=False):
    """分析单步训练显存占用"""
    torch.cuda.reset_peak_memory_stats()
    gc.collect()
    torch.cuda.empty_cache()
    
    # 记录初始状态
    mem_before = get_memory_usage()
    
    # Forward
    outputs = model(**batch)
    logits = outputs.logits
    
    # 计算 loss
    shift_logits = logits[..., :-1, :].contiguous()
    shift_labels = batch['labels'][..., 1:].contiguous()
    loss = torch.nn.functional.cross_entropy(
        shift_logits.view(-1, shift_logits.size(-1)),
        shift_labels.view(-1),
        ignore_index=-100
    )
    
    mem_after_forward = get_memory_usage()
    
    # Backward
    loss.backward()
    mem_after_backward = get_memory_usage()
    
    # Optimizer step
    optimizer.step()
    optimizer.zero_grad()
    mem_after_step = get_memory_usage()
    
    # 汇总
    print(f"显存占用分析 ({'LoRA' if use_lora else '全参'}):")
    print(f"  初始: {mem_before['gpu_allocated']:.2f} GB")
    print(f"  Forward 后: {mem_after_forward['gpu_allocated']:.2f} GB (+{mem_after_forward['gpu_allocated']-mem_before['gpu_allocated']:.2f} GB)")
    print(f"  Backward 后: {mem_after_backward['gpu_allocated']:.2f} GB (+{mem_after_backward['gpu_allocated']-mem_after_forward['gpu_allocated']:.2f} GB)")
    print(f"  Optimizer 后: {mem_after_step['gpu_allocated']:.2f} GB")
    print(f"  峰值: {torch.cuda.max_memory_allocated()/1e9:.2f} GB")
```

## 高频追问（至少 5 个）

### 1. Q: 项目里微调用到了多少显存？如何解释得自洽？

**A**: 以 Qwen2-7B LoRA 微调为例：
- **模型权重**：7B × 2 bytes = 14 GB（FP16，冻结不优化）
- **LoRA 参数**：约 90M 参数 × 2 bytes = 0.18 GB
- **LoRA 优化器状态**：90M × 12 bytes ≈ 1.1 GB（AdamW）
- **激活值**：batch=4, seq=2048, checkpointing → 约 15-20 GB
- **临时缓冲与碎片**：约 3-5 GB
- **总计**：约 35-40 GB（单卡 A100-80GB 可训练）

**自洽性验证**：
- 若回答 "只用 10GB"，需检查是否 batch_size=1 或 gradient_accumulation 很大
- 若回答 "50GB"，需检查是否未开 activation checkpointing
- 若回答 "需要多卡"，需说明是否全参或模型更大

### 2. Q: 14B 模型 FP16 权重大概有多大？训练还要加哪些？INT8 大概能省多少？

**A**: 
- **FP16 权重**：14B × 2 bytes = 28 GB
- **训练额外占用**：
  - 梯度：28 GB
  - 优化器状态（AdamW）：14B × 12 bytes = 168 GB
  - 激活值：取决于 batch 和 seq，约 50-100 GB
  - 总计训练显存：约 280-350 GB（单卡需 ZeRO-3 多卡切分）
- **INT8 节省**：
  - **推理时**：权重 14B × 1 byte = 14 GB，节省 50%
  - **训练时**：主要节省基座权重（28GB→14GB），但优化器状态、梯度仍需 FP16/FP32
  - **QLoRA 训练**：总计可省 10-15 GB（权重部分），但激活值不变

### 3. Q: LoRA 微调 vs 全参微调，显存具体差在哪里？

**A**: 核心差异在**优化器状态和梯度**：

| 项目 | 全参微调 | LoRA 微调 (rank=64) | 差异 |
|------|---------|-------------------|------|
| **模型权重** | 28 GB（可训练） | 28 GB（冻结） | 相同 |
| **梯度** | 28 GB | ~0.2 GB | **节省 99%** |
| **优化器状态** | 168 GB | ~1.2 GB | **节省 99%** |
| **激活值** | 60 GB | 60 GB | **相同**（关键瓶颈！）|
| **总计** | ~320 GB | ~100 GB | 节省 70% |

**关键洞察**：
- LoRA 省的是优化器状态和梯度，但激活值不变
- 若激活值占比高（长序列、大 batch），LoRA 优势缩小
- Activation Checkpointing 对 LoRA 同样重要

### 4. Q: Activation Checkpointing 如何节省显存？代价是什么？

**A**: 
**原理**：
- 前向时不存储所有中间激活，只保留部分"检查点"
- 反向时重新计算检查点之间的激活
- 用计算换显存

**效果**：
- 显存：激活值减少 50-70%
- 速度：训练慢 20-30%（重计算开销）

**配置建议**：
```python
# 全模型检查点（最省显存）
model.gradient_checkpointing_enable()

# 选择性检查点（平衡速度与显存）
# 每隔 k 层保存一次检查点
def selective_checkpoint(layer_idx, k=2):
    return layer_idx % k == 0
```

**适用场景**：
- 显存严重不足时（单卡训大模型）
- 长序列训练（激活值占主导）

### 5. Q: DeepSpeed ZeRO-1/2/3 分别切分什么？如何选择？

**A**: 
| ZeRO Stage | 切分内容 | 显存节省 | 通信开销 | 适用场景 |
|-----------|---------|---------|---------|---------|
| **ZeRO-1** | 优化器状态 | 4× | 低 | 模型中等、显存略紧 |
| **ZeRO-2** | 优化器状态 + 梯度 | 8× | 中 | 模型较大、显存紧张 |
| **ZeRO-3** | 优化器状态 + 梯度 + 参数 | N× (N=卡数) | 高 | 大模型、显存极紧 |

**选择建议**：
- 7B 以下：LoRA 或 ZeRO-1/2
- 7B-30B：ZeRO-2 + Checkpointing
- 30B-70B：ZeRO-3 + Checkpointing
- 70B+：ZeRO-3 + CPU Offload + 模型并行

**注意事项**：
- ZeRO-3 通信开销大，需高带宽网络（InfiniBand）
- CPU Offload 会严重拖慢速度，仅用于调试或极小 batch

### 6. Q: 如何快速估算训练吞吐量（tokens/s）？

**A**: 
**经验公式**：
```
tokens/s ≈ GPU_FLOPS × MFU / (6 × 参数量)
```

**示例（A100-80GB, 312 TFLOPS, MFU=0.4, 7B 模型）**：
```
tokens/s = 312e12 × 0.4 / (6 × 7e9) 
         = 124.8e12 / 42e9
         ≈ 2971 tokens/s
         ≈ 3K tokens/s

每步时间 (batch=4, seq=2048):
= 4 × 2048 / 2971 
≈ 2.76 秒
```

**实际影响因素**：
- MFU（模型算力利用率）：通常 0.3-0.6
- 通信开销：多卡时降低 MFU
- 激活检查点：降低 MFU 约 20%
- 数据加载：需保证不成为瓶颈

### 7. Q: 项目中说"显存不够"时，按什么顺序排查和优化？

**A**: **排查顺序**：
1. **确认瓶颈**：
   ```python
   print(f"参数: {sum(p.numel() for p in model.parameters()) * 2 / 1e9:.2f} GB")
   print(f"激活峰值: {torch.cuda.max_memory_allocated()/1e9:.2f} GB")
   ```

2. **优化策略优先级**：
   - **最优先**：减小 batch_size（直接降激活值）
   - **次优先**：开启 Activation Checkpointing（激活值 -50-70%）
   - **第三**：使用 LoRA（优化器状态 -99%）
   - **第四**：ZeRO-2/3（切分参数和状态）
   - **最后**：CPU Offload（极慢，仅用于验证）

3. **逐步验证**：

## 常见错误（至少 3 个）

### 1. **错误：混淆 LoRA 显存节省的来源**

**错误认知**：LoRA 省的是模型权重（认为冻结权重不占显存）。

**正确理解**：
- LoRA **不省基座模型权重**（仍需 28 GB 存储冻结参数）
- LoRA 省的是**梯度 + 优化器状态**（从 196 GB 降至 1.4 GB）
- **激活值完全相同**，是 LoRA 的瓶颈所在

**验证方法**：
```python
# 错误：只看参数量
print(f"LoRA 参数: {sum(p.numel() for p in model.parameters() if p.requires_grad)}")

# 正确：看各部分占用
mem = get_memory_usage()
print(f"冻结参数: {mem['frozen_params']} GB")
print(f"LoRA 参数: {mem['trainable_params']} GB")
print(f"优化器: {mem['optimizer']} GB")
print(f"激活: {mem['activations']} GB")
```

### 2. **错误：忽略 Activation Checkpointing 的速度代价**

**错误做法**：所有场景都开 checkpointing，不管 batch_size 和序列长度。

**正确做法**：
- **短序列 + 大 batch**：激活值占比低，checkpointing 收益小，不要开
- **长序列 + 小 batch**：激活值占比高，checkpointing 收益大，必须开
- **权衡公式**：
  ```
  时间开销 = 20-30% 额外计算
  显存节省 = 50-70% 激活值
  
  若激活值 > 总显存的 50%，开启值得
  若激活值 < 总显存的 30%，不要开启
  ```

**验证**：
```python
# 不开 checkpointing
batch_size = 4
mem_no_ckpt = train_step(mem_profile=True)  # 记录显存

# 开启 checkpointing
model.gradient_checkpointing_enable()
mem_with_ckpt = train_step(mem_profile=True)

speed_ratio = time_with_ckpt / time_no_ckpt
mem_ratio = mem_with_ckpt / mem_no_ckpt

print(f"速度降低: {(speed_ratio-1)*100:.1f}%")
print(f"显存节省: {(1-mem_ratio)*100:.1f}%")
```

### 3. **错误：ZeRO-3 配置不当导致通信瓶颈**

**错误配置**：
```json
{
  "zero_optimization": {
    "stage": 3,
    "offload_optimizer": {"device": "cpu"},  // CPU offload + ZeRO-3
    "offload_param": {"device": "cpu"}        // 双重 offload
  }
}
```

**问题**：
- ZeRO-3 + CPU Offload 会导致参数频繁在 CPU-GPU 间传输
- PCIe 带宽（32 GB/s）远低于 GPU 显存带宽（2000 GB/s）
- 训练速度降低 3-10 倍

**正确配置**：
```json
{
  "zero_optimization": {
    "stage": 3,
    "offload_optimizer": {"device": "none"},  // 不 offload，靠切分
    "offload_param": {"device": "none"},
    "overlap_comm": true,                      // 通信与计算重叠
    "contiguous_gradients": true
  }
}
```

**何时使用 CPU Offload**：
- 显存极端不足，无法加载模型
- 仅用于调试或极小 batch（batch_size=1）
- 牺牲速度换取可行性

### 4. **错误：INT8 量化训练预期过高**

**错误认知**：INT8 训练可以像推理一样省 50% 显存。

**实际情况**：
- **推理时 INT8**：权重 INT8 + 激活 INT8/FP16，显存省 50%
- **训练时 INT8**：
  - 权重可量化（省一半）
  - 梯度仍需 FP16（反向传播精度）
  - 优化器状态仍需 FP32（动量累积）
  - 激活值通常 FP16（避免梯度爆炸）

**QLoRA 的真正价值**：
```python
# 不是节省训练显存，而是能加载更大模型
# 70B 模型：
# - FP16: 140 GB 权重（需 2×A100-80GB）
# - INT4: 35 GB 权重（单卡 A100-80GB 可加载）

# QLoRA 的优化器状态仍然很小（只优化 LoRA 参数）
# 所以能在单卡上微调 70B 模型
```

### 5. **错误：忽视多卡通信开销**

**错误认知**：多卡线性加速，8 卡就是 8 倍吞吐。

**实际情况**：
- **理想情况**：8 卡 → 8× 吞吐
- **实际情况**：8 卡 → 5-7× 吞吐（通信开销 15-40%）

**通信开销来源**：
- ZeRO-2：梯度 all-reduce（每步一次）
- ZeRO-3：参数 all-gather + 梯度 all-reduce（每步多次）
- TP：每层都需 all-reduce（高频通信）

**优化策略**：
```json
{
  "zero_optimization": {
    "reduce_bucket_size": 5e8,        // 增大分桶，减少通信次数
    "overlap_comm": true,              // 通信与计算重叠
    "contiguous_gradients": true       // 连续内存，减少拷贝
  },
  "gradient_accumulation_steps": 4   // 累积梯度，减少同步频率
}
```

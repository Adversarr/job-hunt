# 主流架构共同点

## 一句话结论
主流大模型架构趋同于 Decoder-only + RMSNorm + SwiGLU + RoPE + GQA，核心驱动是训练效率、推理吞吐和长上下文能力的平衡；MoE 成为趋势源于其能在有限算力下扩展参数规模。

## 核心定义/公式

### 1. Decoder-only 架构标准组件
```
Embedding → N×[RoPE → Multi-Head Attention → RMSNorm → SwiGLU FFN → RMSNorm] → LM Head
```

**RMSNorm（Root Mean Square Layer Normalization）**：
```python
# LayerNorm: 对每个样本计算均值和方差
# RMSNorm: 只计算均方根，省去均值计算，更快且效果相当
def rms_norm(x, weight, eps=1e-6):
    # x: [batch, seq_len, hidden_dim]
    rms = torch.sqrt(torch.mean(x ** 2, dim=-1, keepdim=True) + eps)
    return weight * (x / rms)
```

**SwiGLU（Swish-Gated Linear Unit）**：
```python
# 相比传统 FFN(x) = ReLU(xW1 + b1)W2 + b2
# SwiGLU 引入门控机制，提升表达能力和训练稳定性
def swiglu_ffn(x, W_gate, W_up, W_down, b_gate=None, b_up=None, b_down=None):
    # x: [batch, seq_len, hidden_dim]
    # W_gate, W_up: [hidden_dim, ffn_dim]
    # W_down: [ffn_dim, hidden_dim]
    gate = F.silu(x @ W_gate + b_gate)  # Swish 激活
    up = x @ W_up + b_up
    return (gate * up) @ W_down + b_down
```

**参数量估算**：
- 传统 FFN：2 × d_model × d_ff（通常 d_ff = 4×d_model）
- SwiGLU：3 × d_model × d_ff（多一个门控矩阵，参数量增加 50%）

### 2. RoPE（Rotary Position Embedding）

**核心公式**：
```python
def apply_rotary_pos_emb(q, k, cos, sin):
    # q, k: [batch, heads, seq_len, head_dim]
    # cos, sin: [seq_len, head_dim]
    
    # RoPE 的关键是：位置信息通过旋转矩阵注入
    # q_m = q * cos(mθ) + rotate_half(q) * sin(mθ)
    q_embed = (q * cos) + (rotate_half(q) * sin)
    k_embed = (k * cos) + (rotate_half(k) * sin)
    return q_embed, k_embed

def rotate_half(x):
    # 将向量分成两半并旋转
    x1 = x[..., :x.shape[-1]//2]
    x2 = x[..., x.shape[-1]//2:]
    return torch.cat((-x2, x1), dim=-1)
```

**相对位置编码的优势**：
- 位置 m 和 n 的注意力分数只与 m-n 相关（相对位置）
- 自然支持外推到训练长度之外（配合 scaling 策略）

### 3. GQA（Grouped-Query Attention）

```python
# MHA: 每个头有独立的 Q、K、V
# MQA: 所有头共享一组 K、V（质量下降明显）
# GQA: 将头分组，每组共享 K、V（质量与效率的平衡）

def grouped_query_attention(q, k, v, n_groups=8):
    # q: [batch, n_heads, seq_len, head_dim]
    # k, v: [batch, n_kv_heads, seq_len, head_dim]
    # n_heads 必须能被 n_kv_heads 整除
    
    batch, n_heads, seq_len, head_dim = q.shape
    n_kv_heads = k.shape[1]
    heads_per_group = n_heads // n_kv_heads
    
    # 将 Q 分组，每组共享 K、V
    q = q.view(batch, n_kv_heads, heads_per_group, seq_len, head_dim)
    
    # 扩展 K、V 以匹配 Q 的头数
    k = k.unsqueeze(2).expand(batch, n_kv_heads, heads_per_group, seq_len, head_dim)
    v = v.unsqueeze(2).expand(batch, n_kv_heads, heads_per_group, seq_len, head_dim)
    
    # 标准注意力计算
    scores = torch.matmul(q, k.transpose(-2, -1)) / math.sqrt(head_dim)
    attn_weights = F.softmax(scores, dim=-1)
    output = torch.matmul(attn_weights, v)
    
    return output.reshape(batch, n_heads, seq_len, head_dim)
```

**显存收益**（推理阶段 KV Cache）：
- MHA：KV cache = 2 × n_layers × n_heads × head_dim × seq_len
- GQA：KV cache = 2 × n_layers × n_kv_heads × head_dim × seq_len
- 典型配置（Llama 2 70B）：n_heads=64, n_kv_heads=8，节省 8× 显存

### 4. MoE（Mixture of Experts）

**核心组件**：
```python
class MoELayer(nn.Module):
    def __init__(self, hidden_dim, ffn_dim, n_experts, top_k):
        self.gate = nn.Linear(hidden_dim, n_experts)  # 路由网络
        self.experts = nn.ModuleList([
            SwiGLU_FFN(hidden_dim, ffn_dim // n_experts)
            for _ in range(n_experts)
        ])
        self.top_k = top_k  # 通常 top_k=2
    
    def forward(self, x):
        batch, seq_len, hidden_dim = x.shape
        
        # 1. 路由：为每个 token 选择 top-k 专家
        gate_logits = self.gate(x)  # [batch, seq_len, n_experts]
        topk_values, topk_indices = torch.topk(gate_logits, self.top_k, dim=-1)
        topk_weights = F.softmax(topk_values, dim=-1)  # 归一化权重
        
        # 2. 稀疏激活：只计算被选中的专家
        output = torch.zeros_like(x)
        for i in range(self.top_k):
            expert_idx = topk_indices[..., i]  # [batch, seq_len]
            weight = topk_weights[..., i:i+1]  # [batch, seq_len, 1]
            
            # 路由到对应专家
            for exp_id in range(self.n_experts):
                mask = (expert_idx == exp_id)
                if mask.any():
                    selected_tokens = x[mask]  # [n_selected, hidden_dim]
                    expert_output = self.experts[exp_id](selected_tokens)
                    output[mask] += weight[mask] * expert_output
        
        return output
```

**关键参数**：
- `n_experts`：专家数量（DeepSeek-V2: 160 个，Mixtral-8x7B: 8 个）
- `top_k`：每个 token 激活的专家数（通常 2）
- `capacity_factor`：专家容量因子，控制负载均衡

## 为什么（2-3 个因果链）

### 1. 为什么主流架构收敛到 Decoder-only + RMSNorm + SwiGLU + RoPE？

**原因 1：Decoder-only 天然适配生成任务**
- 现象：主流模型（GPT、LLaMA、Qwen、DeepSeek）都是 Decoder-only
- 根因：因果注意力机制完美契合自回归生成，无需掩码即可保证训练和推理一致性
- 结果：单架构统一训练和推理，工程实现更简单，生态更成熟

**原因 2：RMSNorm + SwiGLU 提升训练效率和稳定性**
- 现象：LayerNorm + ReLU FFN 被 RMSNorm + SwiGLU 替代
- 根因：RMSNorm 计算更快（省去均值计算）；SwiGLU 门控机制提升表达能力和梯度流动
- 结果：训练更快收敛，loss 曲线更平滑，最终效果提升 2-5%（基于 LLaMA 论文）

**原因 3：RoPE 兼顾位置编码效率与外推能力**
- 现象：绝对位置编码（学习式/正弦式）被 RoPE 替代
- 根因：RoPE 的相对位置特性支持长文本外推；无额外参数，不增加序列长度复杂度
- 结果：训练时短文本，推理时可通过 YaRN/NTK 等扩展到 4×-8× 长度

### 2. 为什么 GQA 成为推理优化标配？

**原因 1：KV Cache 显存占用是长文本推理瓶颈**
- 现象：70B 模型在 4K 上下文时 KV Cache 占用超 10GB 显存
- 根因：KV Cache 随序列长度线性增长，MHA 需为每个头存储完整 K、V
- 结果：长上下文场景（16K、32K）显存迅速不足，batch size 严重受限

**原因 2：GQA 在质量损失和显存节省间找到平衡点**
- 现象：MQA（Multi-Query Attention）质量下降 5-10%；MHA 显存占用高
- 根因：MQA 过度压缩导致表示能力损失；GQA 通过分组共享保留多样性
- 结果：GQA（如 Llama 2 70B 用 8 组 KV 头）显存节省 8×，质量损失 < 1%

**原因 3：GQA 与 Paged Attention、FlashAttention 兼容**
- 现象：vLLM、TensorRT-LLM 等推理框架全面支持 GQA
- 根因：GQA 的 KV 复用特性天然适配 cache 优化和批处理
- 结果：工程落地成本低，无需额外适配

### 3. 为什么 MoE 架构成为大模型趋势？

**原因 1：算力约束下扩展参数规模的唯一路径**
- 现象：Dense 70B 模型训练成本已接近主流算力上限；MoE 可达 500B+ 参数
- 根因：MoE 稀疏激活，每个 token 只计算 top-k 专家，实际计算量 ≈ Dense 的 1/8
- 结果：同等算力下，MoE 参数量可达 Dense 的 5-10×，性能显著提升

**原因 2：专家分化提升特定领域能力**
- 现象：不同专家倾向于处理不同类型的 token（代码、数学、自然语言）
- 根因：路由网络学会将相似 token 分配到相同专家，专家逐渐专业化
- 结果：MoE 在多任务、多领域场景下泛化能力更强

**原因 3：推理成本可控**
- 现象：MoE 推理速度与激活参数量相关，而非总参数量
- 根因：每个 token 只激活 top-k 专家（如 2/160），实际计算量 ≈ 激活参数的 Dense 模型
- 结果：DeepSeek-V2（236B 总参数，21B 激活）推理成本接近 Dense 21B 模型

### 4. 预训练时为什么要用 Masked LM？

**原因 1：双向上下文理解能力**
- 现象：BERT 使用 MLM，GPT 使用 CLM（Causal LM）；理解任务 BERT 效果更好
- 根因：MLM 允许模型同时看到被 mask token 的左右上下文，学习双向依赖
- 结果：分类、NER、抽取等理解类任务，MLM 预训练效果优于 CLM

**原因 2：数据效率更高**
- 现象：MLM 通常只 mask 15% token；CLM 需预测所有 token
- 根因：MLM 每个样本可提供 15% 的监督信号，CLM 每个 token 都是监督，但信息冗余
- 结果：同等数据量下，MLM 学习效率更高（但 CLM 更适合生成任务）

**原因 3：生成任务的因果性约束**
- 现象：主流大模型（GPT、LLaMA）预训练都用 CLM 而非 MLM
- 根因：生成任务要求预测下一个 token 时不能看到未来信息；MLM 破坏了因果性
- 结果：CLM 天然适配生成任务，预训练和推理行为一致；MLM 预训练需要额外适配

**Step 2：实现 KV Cache 复用**
```python
# 推理时 KV Cache 管理
class KVCache:
    def __init__(self, n_layers, n_kv_heads, head_dim, max_seq_len):
        self.cache = torch.zeros(n_layers, 2, n_kv_heads, max_seq_len, head_dim)
    
    def update(self, layer_idx, k, v, start_pos):
        # k, v: [batch, n_kv_heads, seq_len, head_dim]
        self.cache[layer_idx, 0, :, start_pos:start_pos+seq_len, :] = k
        self.cache[layer_idx, 1, :, start_pos:start_pos+seq_len, :] = v
    
    def get(self, layer_idx, start_pos, end_pos):
        k = self.cache[layer_idx, 0, :, start_pos:end_pos, :]
        v = self.cache[layer_idx, 1, :, start_pos:end_pos, :]
        return k, v
```

#### 3. 搭建 MoE 架构

**Step 1：确定专家数量和拓扑**

# 小模型 MoE（如 Mixtral-8x7B）
n_experts = 8

expert_hidden = 14336  # 每个专家的 FFN 维度

# 大模型 MoE（如 DeepSeek-V2）
n_experts = 160

shared_experts = 2  # 共享专家（所有 token 都走）

**Step 2：实现负载均衡 loss**
```python
def load_balancing_loss(gate_logits, topk_indices, n_experts):
    """
    gate_logits: [batch, seq_len, n_experts]
    topk_indices: [batch, seq_len, top_k]
    """
    batch, seq_len, _ = gate_logits.shape
    
    # 1. 计算每个专家的路由概率均值
    router_probs = F.softmax(gate_logits, dim=-1)  # [batch, seq_len, n_experts]
    router_probs_mean = router_probs.mean(dim=[0, 1])  # [n_experts]
    
    # 2. 计算每个专家的实际负载比例
    expert_counts = torch.zeros(n_experts, device=gate_logits.device)
    for idx in topk_indices.flatten():
        expert_counts[idx] += 1
    expert_load = expert_counts / (batch * seq_len * top_k)
    
    # 3. 负载均衡 loss：鼓励路由概率和负载接近
    aux_loss = n_experts * torch.sum(router_probs_mean * expert_load)
    
    return aux_loss
```

**Step 3：处理专家容量限制**
```python
# 专家容量：每个专家最多处理的 token 数
# capacity = (batch * seq_len * top_k / n_experts) * capacity_factor
# capacity_factor 通常设为 1.0-2.0

def route_with_capacity(x, gate_logits, top_k, capacity_factor=1.25):
    batch, seq_len, hidden = x.shape
    n_experts = gate_logits.shape[-1]
    
    # 计算专家容量
    capacity = int((batch * seq_len * top_k / n_experts) * capacity_factor)
    
    # 路由并限制容量
    topk_values, topk_indices = torch.topk(gate_logits, top_k, dim=-1)
    expert_counts = torch.zeros(n_experts, device=x.device)
    
    routed_tokens = []
    for i in range(batch):
        for j in range(seq_len):
            for k in range(top_k):
                expert_idx = topk_indices[i, j, k].item()
                if expert_counts[expert_idx] < capacity:
                    routed_tokens.append((i, j, expert_idx, topk_values[i, j, k]))
                    expert_counts[expert_idx] += 1
    
    return routed_tokens
```

#### 4. 预训练目标选择

**Step 1：明确任务类型**
- 纯生成任务（对话、续写）：CLM（Causal LM）
- 理解+生成任务（翻译、摘要）：CLM 或 Encoder-Decoder + MLM
- 理解任务（分类、抽取）：MLM（如 BERT）或 CLM + 指令微调

**Step 2：实现 CLM Loss**
```python
def causal_lm_loss(logits, labels, ignore_index=-100):
    """
    logits: [batch, seq_len, vocab_size]
    labels: [batch, seq_len]
    """
    # 移位：预测下一个 token
    shift_logits = logits[..., :-1, :].contiguous()
    shift_labels = labels[..., 1:].contiguous()
    
    # 交叉熵损失
    loss = F.cross_entropy(
        shift_logits.view(-1, shift_logits.size(-1)),
        shift_labels.view(-1),
        ignore_index=ignore_index
    )
    return loss
```

**Step 3：实现 MLM Loss（可选）**
```python
def masked_lm_loss(logits, labels, mask_positions):
    """
    logits: [batch, seq_len, vocab_size]
    labels: [batch, seq_len]
    mask_positions: [batch, seq_len]，布尔掩码，True 表示被 mask 的位置
    """
    # 只计算被 mask 位置的损失
    loss = F.cross_entropy(
        logits[mask_positions],
        labels[mask_positions]
    )
    return loss
```

### 代码示例

**完整的 Decoder-only Layer 实现**：
```python
import torch
import torch.nn as nn
import torch.nn.functional as F
import math

class RMSNorm(nn.Module):
    def __init__(self, hidden_dim, eps=1e-6):
        super().__init__()
        self.weight = nn.Parameter(torch.ones(hidden_dim))
        self.eps = eps
    
    def forward(self, x):
        # x: [batch, seq_len, hidden_dim]
        rms = torch.sqrt(torch.mean(x ** 2, dim=-1, keepdim=True) + self.eps)
        return self.weight * (x / rms)

class RotaryEmbedding(nn.Module):
    def __init__(self, head_dim, max_seq_len=2048, base=10000):
        super().__init__()
        # 计算频率
        inv_freq = 1.0 / (base ** (torch.arange(0, head_dim, 2).float() / head_dim))
        self.register_buffer("inv_freq", inv_freq)
        
        # 预计算 cos 和 sin
        pos = torch.arange(max_seq_len).float()
        freqs = torch.einsum("i,j->ij", pos, inv_freq)  # [seq_len, head_dim//2]
        emb = torch.cat([freqs, freqs], dim=-1)  # [seq_len, head_dim]
        self.register_buffer("cos_cached", emb.cos()[None, None, :, :])
        self.register_buffer("sin_cached", emb.sin()[None, None, :, :])
    
    def forward(self, seq_len):
        return self.cos_cached[:, :, :seq_len, :], self.sin_cached[:, :, :seq_len, :]

def apply_rotary_pos_emb(q, k, cos, sin):
    # q, k: [batch, n_heads, seq_len, head_dim]
    q_embed = (q * cos) + (rotate_half(q) * sin)
    k_embed = (k * cos) + (rotate_half(k) * sin)
    return q_embed, k_embed

def rotate_half(x):
    x1 = x[..., :x.shape[-1]//2]
    x2 = x[..., x.shape[-1]//2:]
    return torch.cat([-x2, x1], dim=-1)

class GroupedQueryAttention(nn.Module):
    def __init__(self, hidden_dim, n_heads, n_kv_heads):
        super().__init__()
        self.n_heads = n_heads
        self.n_kv_heads = n_kv_heads
        self.head_dim = hidden_dim // n_heads
        self.heads_per_group = n_heads // n_kv_heads
        
        self.q_proj = nn.Linear(hidden_dim, n_heads * self.head_dim, bias=False)
        self.k_proj = nn.Linear(hidden_dim, n_kv_heads * self.head_dim, bias=False)
        self.v_proj = nn.Linear(hidden_dim, n_kv_heads * self.head_dim, bias=False)
        self.o_proj = nn.Linear(n_heads * self.head_dim, hidden_dim, bias=False)
        
        self.rotary_emb = RotaryEmbedding(self.head_dim)
    
    def forward(self, x, kv_cache=None, start_pos=0):
        batch, seq_len, _ = x.shape
        
        # 投影
        q = self.q_proj(x).view(batch, seq_len, self.n_heads, self.head_dim).transpose(1, 2)
        k = self.k_proj(x).view(batch, seq_len, self.n_kv_heads, self.head_dim).transpose(1, 2)
        v = self.v_proj(x).view(batch, seq_len, self.n_kv_heads, self.head_dim).transpose(1, 2)
        
        # 应用 RoPE
        cos, sin = self.rotary_emb(start_pos + seq_len)
        q, k = apply_rotary_pos_emb(q, k, cos, sin)
        
        # 更新 KV Cache
        if kv_cache is not None:
            k, v = kv_cache.update(k, v, start_pos)
        
        # GQA：扩展 KV 头以匹配 Q 头
        k = k.unsqueeze(2).expand(batch, self.n_kv_heads, self.heads_per_group, -1, self.head_dim)
        v = v.unsqueeze(2).expand(batch, self.n_kv_heads, self.heads_per_group, -1, self.head_dim)
        k = k.reshape(batch, self.n_heads, -1, self.head_dim)
        v = v.reshape(batch, self.n_heads, -1, self.head_dim)
        
        # 注意力计算
        scores = torch.matmul(q, k.transpose(-2, -1)) / math.sqrt(self.head_dim)
        attn_weights = F.softmax(scores, dim=-1)
        output = torch.matmul(attn_weights, v)
        
        # 输出投影
        output = output.transpose(1, 2).reshape(batch, seq_len, -1)
        return self.o_proj(output)

class SwiGLU(nn.Module):
    def __init__(self, hidden_dim, intermediate_dim):
        super().__init__()
        self.gate_proj = nn.Linear(hidden_dim, intermediate_dim, bias=False)
        self.up_proj = nn.Linear(hidden_dim, intermediate_dim, bias=False)
        self.down_proj = nn.Linear(intermediate_dim, hidden_dim, bias=False)
    
    def forward(self, x):
        return self.down_proj(F.silu(self.gate_proj(x)) * self.up_proj(x))

class TransformerBlock(nn.Module):
    def __init__(self, hidden_dim, n_heads, n_kv_heads, intermediate_dim):
        super().__init__()
        self.attention = GroupedQueryAttention(hidden_dim, n_heads, n_kv_heads)
        self.feed_forward = SwiGLU(hidden_dim, intermediate_dim)
        self.attention_norm = RMSNorm(hidden_dim)
        self.ffn_norm = RMSNorm(hidden_dim)
    
    def forward(self, x, kv_cache=None, start_pos=0):
        # Pre-norm + 残差
        h = x + self.attention(self.attention_norm(x), kv_cache, start_pos)
        out = h + self.feed_forward(self.ffn_norm(h))
        return out

class LlamaModel(nn.Module):
    def __init__(self, config):
        super().__init__()
        self.embed_tokens = nn.Embedding(config["vocab_size"], config["hidden_dim"])
        self.layers = nn.ModuleList([
            TransformerBlock(
                config["hidden_dim"],
                config["n_heads"],
                config["n_kv_heads"],
                config["intermediate_size"]
            )
            for _ in range(config["n_layers"])
        ])
        self.norm = RMSNorm(config["hidden_dim"])
        self.lm_head = nn.Linear(config["hidden_dim"], config["vocab_size"], bias=False)
        
        # 权重共享
        self.lm_head.weight = self.embed_tokens.weight
    
    def forward(self, input_ids, kv_cache=None, start_pos=0):
        h = self.embed_tokens(input_ids)
        
        for i, layer in enumerate(self.layers):
            layer_cache = kv_cache[i] if kv_cache else None
            h = layer(h, layer_cache, start_pos)
        
        h = self.norm(h)
        logits = self.lm_head(h)
        return logits
```

## 权衡分析

| 方案 | 收益 | 代价 | 适用边界 |
|------|------|------|----------|
| **MHA（全头独立）** | 表达能力最强，质量最优 | KV Cache 显存占用高（O(n_heads)），长文本推理成本大 | 小模型（<7B）、短上下文（<4K）、质量优先场景 |
| **GQA（分组共享）** | KV Cache 减少 2-8×，质量损失 <1% | 实现复杂度增加，表示能力略有下降 | 大模型（≥7B）、长上下文（≥8K）、推理优先场景 |
| **MQA（全头共享）** | KV Cache 减少最多（n_heads×），推理最快 | 质量下降 5-10%，长序列更明显 | 极端推理优化场景、可接受质量损失 |
| **Dense FFN** | 训练稳定，实现简单 | 参数量大，计算成本高 | 小模型、算力充足、训练优先 |
| **MoE FFN** | 参数量扩展 5-10×，计算量可控 | 训练不稳定（负载均衡）、通信开销大、路由复杂 | 大模型、算力有限、追求极致性能 |
| **LayerNorm** | 训练稳定，归一化完整 | 计算稍慢（需计算均值和方差） | 小模型、训练不稳定场景 |
| **RMSNorm** | 计算更快（省去均值），效果相当 | 归一化信息略少 | 大模型、训练优先、生态成熟（LLaMA） |
| **传统 FFN（ReLU）** | 参数少、计算快 | 表达能力弱，训练收敛慢 | 小模型、推理优先 |
| **SwiGLU FFN** | 表达能力强、收敛快、质量优 | 参数量增加 50%、计算稍慢 | 大模型、质量优先、主流选择 |
| **学习式位置编码** | 实现简单，可学习 | 不支持外推、参数多、序列长度固定 | 固定长度任务、小模型 |
| **RoPE** | 支持外推、相对位置、无额外参数 | 长距离依赖可能衰减、需要 scaling 策略 | 主流选择、长上下文任务 |
| **CLM（Causal LM）** | 天然适配生成、训练推理一致 | 单向上下文、理解任务需微调 | 生成任务、主流大模型 |
| **MLM（Masked LM）** | 双向上下文、理解能力强、数据效率高 | 不适配生成任务、预训练推理不一致 | 理解任务、BERT 类模型 |

## 高频追问（至少 5 个）

### 1. Q: Decoder-only 和 Encoder-Decoder 有什么本质区别？什么时候用 Encoder-Decoder？

**A:** 
- **本质区别**：
  - Decoder-only：因果注意力，只能看到当前位置之前的 token，天然适配自回归生成
  - Encoder-Decoder：Encoder 双向注意力（可见全部输入），Decoder 单向注意力，编码器和解码器通过 cross-attention 交互
  
- **使用场景**：
  - Decoder-only：对话、续写、代码生成、单任务微调（当前主流）
  - Encoder-Decoder：翻译、摘要、问答（需要理解输入再生成输出的任务，如 T5、BART）
  
- **为什么 Decoder-only 成为主流**：
  1. 架构统一：预训练和推理行为一致
  2. 生态成熟：LLaMA、Qwen、DeepSeek 全线 Decoder-only
  3. 指令微调：通过 SFT 可让 Decoder-only 模型学会理解+生成
  4. 工程简单：无需维护两套注意力机制和缓存

### 2. Q: GQA 相比 MHA 为什么质量损失小？如何选择 n_kv_heads？

**A:**
- **质量损失小的原因**：
  1. **冗余性**：不同头的注意力模式有重叠，相邻头学习的表示相似
  2. **保留多样性**：分组共享保留了组间差异，质量接近 MHA
  3. **实验验证**：LLaMA 2 70B 用 8 组 KV 头，性能下降 < 1%
  
- **n_kv_heads 选择**：
  - **经验规则**：`n_kv_heads = n_heads // g`，g 通常为 2-8
  - **小模型（<7B）**：g=2（如 32 头用 16 组 KV），平衡质量与显存
  - **大模型（≥7B）**：g=8（如 64 头用 8 组 KV），显存节省更关键
  - **极致优化**：g=n_heads（MQA），用于边缘设备推理
  
- **实际落地**：
  ```python
  # Llama 2 7B: 32 头，32 组 KV（MHA）
  # Llama 2 13B: 40 头，40 组 KV（MHA）
  # Llama 2 70B: 64 头，8 组 KV（GQA）
  # Qwen-14B: 40 头，40 组 KV（MHA）
  # Qwen-72B: 64 头，8 组 KV（GQA）
  ```

### 3. Q: MoE 的路由网络如何训练？负载不均衡怎么办？

**A:**
- **路由网络训练**：
  1. **联合训练**：路由网络（gate）和专家网络一起训练，通过反向传播优化
  2. **路由策略**：选择 top-k 专家，权重通过 softmax 归一化
  3. **梯度传播**：只有被选中的专家参与反向传播，未选中专家梯度为 0
  
- **负载不均衡问题**：
  1. **现象**：部分专家被频繁调用，部分专家几乎不使用（"坍塌"）
  2. **原因**：路由网络可能学会"偷懒"，总选少数专家
  
- **解决方案**：
  ```python
  # 方案1：负载均衡 loss
  aux_loss = n_experts * sum(router_prob_mean * expert_load)
  total_loss = task_loss + aux_loss_weight * aux_loss
  
  # 方案2：专家容量限制
  capacity = (batch * seq_len * top_k / n_experts) * capacity_factor
  # 超过容量的 token 丢弃或路由到其他专家
  
  # 方案3：噪声路由（训练时）
  gate_logits = gate_logits + torch.randn_like(gate_logits) * noise_std
  
  # 方案4：共享专家（DeepSeek-V2）
  # 固定部分专家（如 2 个）所有 token 都走，保证基础能力
  ```

### 4. Q: RMSNorm 和 LayerNorm 在大模型训练中的差异是什么？

**A:**
- **计算差异**：
  ```python
  # LayerNorm: 计算均值和方差
  mean = x.mean(dim=-1, keepdim=True)
  var = x.var(dim=-1, keepdim=True)
  x_norm = (x - mean) / sqrt(var + eps)
  
  # RMSNorm: 只计算均方根
  rms = sqrt(mean(x^2) + eps)
  x_norm = x / rms
  ```
  
- **性能差异**：
  1. **计算速度**：RMSNorm 快约 10-20%（省去均值计算）
  2. **显存占用**：RMSNorm 略低（不存储均值）
  3. **训练稳定性**：两者相当，LayerNorm 在小模型稍稳
  
- **为什么 RMSNorm 成为主流**：
  1. LLaMA 系列使用，生态成熟
  2. 计算效率在大模型训练中更重要
  3. 实验表明：大模型对归一化方式不敏感，RMSNorm 效果足够好
  
- **何时用 LayerNorm**：
  - 小模型（<1B）训练不稳定时
  - 需要严格均值归一化的场景（如对抗训练）
  - 兼容旧模型生态（如 BERT、GPT-2）

### 5. Q: SwiGLU 相比传统 FFN 有什么优势？参数量增加了多少？

**A:**
- **SwiGLU 结构**：
  ```python
  # 传统 FFN
  output = ReLU(x @ W1) @ W2
  
  # SwiGLU FFN
  output = (SiLU(x @ W_gate) * (x @ W_up)) @ W_down
  ```
  
- **优势**：
  1. **门控机制**：通过 SiLU(x @ W_gate) 控制信息流动，自适应选择激活路径
  2. **非单调性**：SiLU（x * sigmoid(x)）非单调，比 ReLU 表达能力更强
  3. **梯度流动**：门控机制改善梯度流动，训练更稳定
  4. **实验验证**：LLaMA 论文表明 SwiGLU 比传统 FFN 效果提升 2-5%
  
- **参数量增加**：
  ```python
  # 传统 FFN: 2 个矩阵
  params = hidden_dim * ffn_dim * 2
  
  # SwiGLU: 3 个矩阵（W_gate, W_up, W_down）
  params = hidden_dim * ffn_dim * 3
  
  # 增加比例: 3/2 = 1.5 倍（增加 50%）
  ```
  
- **权衡**：
  - 收益：质量提升 2-5%，训练更快收敛
  - 代价：参数量增加 50%，推理计算量增加 33%
  - 结论：大模型算力相对充足，质量收益大于计算成本，值得使用

### 6. Q: 为什么 RoPE 能支持长文本外推？如何从 4K 扩展到 16K？

**A:**
- **RoPE 外推原理**：
  1. **相对位置**：RoPE 编码的是相对位置 m-n，而非绝对位置
  2. **频率衰减**：高频分量关注近邻位置，低频分量关注远距离位置
  3. **外推能力**：低频分量对位置变化不敏感，可泛化到训练长度外
  
- **外推方法（4K → 16K）**：
  ```python
  # 方法1：线性插值（Linear Scaling）
  # 将位置压缩到训练范围内
  scale = original_seq_len / target_seq_len  # 4096 / 16384 = 0.25
  pos_scaled = pos * scale
  
  # 方法2：NTK-Aware Scaling
  # 调整频率基数，保持高频分量不变
  base_new = base * (scale ** (dim / (dim - 2)))
  
  # 方法3：YaRN（Yet another RoPE extension）
  # 结合插值和温度调整
  # 对不同频率分量应用不同策略
  ```
  
- **实际落地（vLLM 示例）**：
  ```python
  from vllm import LLM
  
  # 加载 4K 模型，推理 16K
  llm = LLM(
      model="meta-llama/Llama-2-7b-hf",
      max_model_len=16384,  # 扩展到 16K
      rope_scaling={
          "type": "linear",
          "factor": 4.0  # 16K / 4K
      }
  )
  ```
  
- **副作用**：
  1. 长距离依赖可能减弱（需要 scaling 调优）
  2. 短文本性能可能轻微下降（YaRN 优化了这点）
  3. 需要验证：外推后需在长文本上测试质量

### 7. Q: MoE 推理时如何保证负载均衡？动态路由会带来什么问题？

**A:**
- **推理时负载均衡**：
  1. **容量限制**：每个专家设置最大处理 token 数（capacity）
  2. **溢出处理**：超过容量的 token 路由到其他专家或丢弃
  3. **静态分配**：部分模型推理时使用固定路由（训练时动态，推理时静态）
  
- **动态路由问题**：
  1. **批处理困难**：不同 token 走不同专家，无法并行计算
  2. **显存碎片**：专家输出需重新组合，产生临时缓存
  3. **延迟抖动**：路由不均衡导致专家负载波动，影响 P99 延迟
  
- **优化策略**：
  ```python
  # 策略1：Token 分组批处理
  # 将走相同专家的 token 分组，批量计算
  
  # 策略2：专家并行（Expert Parallelism）
  # 不同专家分布在不同 GPU，路由后发送 token
  
  # 策略3：共享专家（DeepSeek-V2）
  # 固定部分专家（如 2/160），所有 token 都走，保证基础计算
  
  # 策略4：路由缓存
  # 对相似输入缓存路由结果（需验证一致性）
  ```

## 常见错误（至少 3 个）

### 1. 错误：认为 GQA 是 MQA 的改进版，质量一定比 MHA 差

**错误原因**：
- 忽视了不同规模模型的权衡差异
- 未考虑实际质量损失（<1%）远小于显存收益（8×）

**正确理解**：
- GQA 是 MHA（质量优先）和 MQA（速度优先）的平衡点
- 大模型（≥70B）头数多，冗余大，GQA 质量损失极小
- 实测：Llama 2 70B（GQA）vs 原始 70B（MHA）性能几乎一致
- 选择依据：
  - 小模型（<7B）：MHA 或 GQA（g=2）
  - 大模型（≥7B）：GQA（g=8），长上下文必选

### 2. 错误：认为 MoE 训练和推理成本相同

**错误原因**：
- 混淆"参数量"和"计算量"
- 忽视稀疏激活的特性

**正确理解**：
- **训练**：需要路由计算、负载均衡、通信开销，总成本高于 Dense
- **推理**：激活参数决定计算量，MoE 推理成本远低于总参数量对应的 Dense 模型
- **示例**：
  - DeepSeek-V2：236B 总参数，21B 激活参数
  - 推理速度 ≈ Dense 21B 模型（实际稍慢，因路由开销）
  - 训练成本 > Dense 236B 模型（因路由和通信）
- **结论**：MoE 适合"推理密集、训练算力有限"的场景

### 3. 错误：混淆 CLM 和 MLM 的应用场景

**错误原因**：
- 只看到 MLM 的数据效率优势
- 忽视生成任务的因果性约束

**正确理解**：
- **CLM（Causal LM）**：
  - 适用：生成任务（对话、续写、代码生成）
  - 训练：预测下一个 token，因果掩码
  - 推理：与训练行为完全一致
  - 主流：GPT、LLaMA、Qwen、DeepSeek
  
- **MLM（Masked LM）**：
  - 适用：理解任务（分类、NER、抽取）
  - 训练：mask 15% token，双向预测
  - 推理：需要特殊适配（如 [CLS] token、encoder-only）
  - 主流：BERT、RoBERTa
  
- **关键洞察**：
  - Decoder-only + CLM 可通过 SFT 学会理解任务
  - Encoder-Decoder + MLM 不适合自回归生成
  - 当前趋势：统一用 Decoder-only，通过数据和指令微调覆盖所有任务

### 4. 错误：认为 RoPE 只是一种位置编码，和绝对位置编码一样

**错误原因**：
- 只关注 RoPE 的实现，未理解其相对位置特性
- 忽视外推能力的关键区别

**正确理解**：
- **绝对位置编码**（学习式、正弦式）：
  - 每个位置有唯一编码，位置信息独立
  - 无法泛化到训练长度外（如训练 4K，推理 8K 时位置编码未见过）
  - 需要外推时效果急剧下降
  
- **RoPE（相对位置编码）**：
  - 位置 m 和 n 的注意力分数只依赖 m-n（相对位置）
  - 高频分量关注近邻，低频分量关注远距离
  - 外推能力：可通过 scaling（线性插值、NTK、YaRN）扩展到 4×-8× 长度
  
- **实际验证**：
  ```python
  # Llama 2 训练 4K，推理 16K
  # RoPE + Linear Scaling: 可用
  # 绝对位置编码: 失败（位置编码未见过）
  ```

### 5. 错误：认为 SwiGLU 只是换了激活函数，不重要

**错误原因**：
- 低估门控机制的表达能力
- 忽视参数量增加的影响和收益

**正确理解**：
- **SwiGLU vs ReLU FFN**：
  - 参数量增加 50%（3 矩阵 vs 2 矩阵）
  - 质量提升 2-5%（LLaMA 论文）
  - 训练收敛更快、更稳定
  
- **为什么重要**：
  1. 大模型每个组件的优化都很关键（边际收益累积）
  2. SwiGLU 已成为 LLaMA、Qwen、Mistral 等主流模型的标配
  3. 门控机制的自适应性提升模型对复杂模式的学习能力
  
- **工程启示**：
  - 实现时注意 3 个投影矩阵的初始化
  - 推理时可量化中间激活（gate 和 up 输出）以节省显存
  - 与 RMSNorm 配合效果更好（Pre-norm 结构）
